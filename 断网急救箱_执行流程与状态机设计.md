### 1.2 执行顺序与并行策略

PS 5.1 无 `ForEach-Object -Parallel`，使用 `Start-Job` 实现并行。

| 阶段 | 模块 | 并行组 | 触发条件 | 说明 |
|------|------|--------|----------|------|
| **Phase 0** | M1, M5, M8, M9 | 4 个 `Start-Job` 并行 | 始终执行 | 无依赖，直接启动 |
| **Phase 1** | M2 | 串行 | M1 完成后 | 需要 M1 的适配器索引/名称 |
| **Phase 2** | M3, M4, M6 | 3 个 `Start-Job` 并行 | M2 完成后 | 均依赖 M2 的 IP/网关/DNS 信息 |
| **Phase 3** | M7 | 串行 | Phase 2 全部完成 | 协议栈检测需综合前序结果，且 `netsh` 命令互斥（不能与其他 netsh 并行） |

**关键约束**：所有 `netsh` 子命令共享全局锁，不可并行。诊断阶段的 netsh 调用也需要串行化控制，但诊断用 `netsh show` 类命令较轻量，Phase 3 单独隔离。

### 1.3 每模块详情

#### M1 — 网络适配器检测

| 属性 | 值 |
|------|-----|
| **超时** | 800ms |
| **依赖** | 无 |
| **并行组** | Phase 0 |
| **核心检测项** | `Get-NetAdapter` 枚举所有适配器；APIPA 地址（`169.254.x.x`）关联 `Status -eq 'Up'`；媒体断开；驱动状态 |
| **APIPA 判断逻辑** | 仅当适配器 `Status -eq 'Up'` 且 IP 为 `169.254.0.0/16` 时标记为 APIPA 故障；`Disabled` 或 `Disconnected` 的适配器上出现 APIPA 不算故障 |
| **降级策略** | `Get-NetAdapter` 失败 → 回退为 `Get-WmiObject Win32_NetworkAdapter` (较慢但兼容性更好)；仍失败 → 标记 M1=UNKNOWN, 跳过 M2 适配器绑定，M2 直接 FAIL |

#### M2 — IP 配置检查

| 属性 | 值 |
|------|-----|
| **超时** | 600ms |
| **依赖** | M1（需要活跃适配器的 `ifIndex` 或 `Name`） |
| **并行组** | Phase 1（串行，但为 M3/M4/M6 的汇聚点） |
| **核心检测项** | `Get-NetIPAddress` + `Get-NetIPConfiguration` 获取 IP/掩码/网关/DNS/DHCP 状态；IPv4 与 IPv6 独立评估 |
| **降级策略** | NetAdapter 模块不可用 → 回退 `ipconfig /all` 文本解析；完全失败 → M3/M4/M6 标记 SKIP |

#### M3 — DNS 解析测试

| 属性 | 值 |
|------|-----|
| **超时** | 1200ms |
| **依赖** | M2（需要 DNS 服务器地址列表） |
| **并行组** | Phase 2 |
| **核心检测项** | 主：`Resolve-DnsName` 解析固定名单（`www.baidu.com`, `www.microsoft.com`, `dns.google`）；辅：`Test-Connection -Count 1` Ping 作为 ICMP 可达性参考。**ICMP 可能被防火墙阻止，Ping 失败不直接判 DNS 故障**——仅当 Resolve-DnsName 也失败才标记 DNS 故障 |
| **降级策略** | `Resolve-DnsName` 不可用 → 仅用 `[System.Net.Dns]::GetHostEntry()` 替代；完全失败 → M3=UNKNOWN |

#### M4 — 网关可达性

| 属性 | 值 |
|------|-----|
| **超时** | 1000ms |
| **依赖** | M2（需要默认网关 IP） |
| **并行组** | Phase 2 |
| **核心检测项** | `Test-Connection -Count 2` Ping 网关；ARP 表检查（`Get-NetNeighbor`）确认网关 MAC 解析正常。若无默认网关（`0.0.0.0`）→ 标记路由缺失 |
| **降级策略** | `Test-Connection` 超时 → 降级为 ARP 表+本地路由表判断；完全失败 → M4=UNKNOWN |

#### M5 — 第三方安全软件残留

| 属性 | 值 |
|------|-----|
| **超时** | 1500ms |
| **依赖** | 无 |
| **并行组** | Phase 0 |
| **核心检测项** | 4 个子检测，`Start-Job` 内并行：
| (a) **注册表驱动扫描** | `HKLM\SYSTEM\CurrentControlSet\Services` 枚举子键，匹配已知驱动名：`360*`, `ZhuDongFangYu`, `hrwfpdrv`, `hrdevmon`, `HipsDaemon`, `TesSafe`, `QQSysMonX64` 等；检测 `Start` 值 ≠ 4（禁用）即为活跃 |
| (b) **Winsock LSP 目录** | 检查 `HKLM\SYSTEM\CurrentControlSet\Services\WinSock2\Parameters\Protocol_Catalog9\Catalog_Entries` 子键数量是否在正常范围（Win10/11 通常 12-20 个）；枚举每个 LSP 的 `PackedCatalogItem` 提取 DLL 路径，校验文件是否存在、数字签名是否有效 |
| (c) **WFP 过滤器** | `netsh wfp show filters` 输出解析，检测非 Microsoft 发布的过滤器（按 `provider.name` 字段判断）；匹配已知第三方 provider 名 |
| (d) **TAP/Wintun 虚拟适配器** | `Get-NetAdapter` 检测 Description 包含 `TAP`, `Wintun`, `VirtualBox`, `VMware` 且状态异常（残留在系统中的虚拟网卡驱动） |
| **降级策略** | 任一子项超时 → 子项标记 TIMEOUT，不影响其他；注册表权限不足 → M5 标记 PARTIAL；`netsh wfp` 失败 → WFP 子项标记 SKIP |

#### M6 — VPN/代理残留

| 属性 | 值 |
|------|-----|
| **超时** | 800ms |
| **依赖** | M2（需要路由表信息判断 VPN 残留路由） |
| **并行组** | Phase 2 |
| **核心检测项** | WinHTTP 代理（`netsh winhttp show proxy`）；IE 代理（注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Internet Settings` 的 `ProxyEnable`/`ProxyServer`）；系统代理（`Get-ItemProperty` 同路径）；VPN 残留路由（`Get-NetRoute` 中目标为 `0.0.0.0/0` 且下一跳非正常网关的条目、Metric 异常低的条目） |
| **降级策略** | 注册表读取失败 → 仅检查 WinHTTP；完全失败 → M6=UNKNOWN |

#### M7 — Winsock/TCP-IP 协议栈健康

| 属性 | 值 |
|------|-----|
| **超时** | 1000ms |
| **依赖** | 前序所有模块完成（综合判断） |
| **并行组** | Phase 3（串行，netsh 命令互斥） |
| **核心检测项** | `netsh int ip show global` 解析任务卸载状态异常；`netsh int tcp show global` 检查 Chimney/RSS/自动调谐是否被恶意关闭；`netsh winsock show catalog` 快速枚举计数（不解析详情，详情由 M5 负责）；TCP 端口绑定异常检测（`Get-NetTCPConnection` 检查 `0.0.0.0:0` 监听） |
| **降级策略** | netsh 命令全部失败 → M7=UNKNOWN（通常意味着协议栈严重损坏） |

#### M8 — 防火墙状态

| 属性 | 值 |
|------|-----|
| **超时** | 500ms |
| **依赖** | 无 |
| **并行组** | Phase 0 |
| **核心检测项** | `Get-NetFirewallProfile` 检查 Domain/Private/Public 三配置文件的 `Enabled` 状态和 `DefaultInboundAction`/`DefaultOutboundAction`；防火墙服务运行状态（`Get-Service MpsSvc`） |
| **降级策略** | Cmdlet 不可用 → 回退 `netsh advfirewall show allprofiles` 文本解析；完全失败 → M8=UNKNOWN |

#### M9 — Hosts 文件劫持

| 属性 | 值 |
|------|-----|
| **超时** | 300ms |
| **依赖** | 无 |
| **并行组** | Phase 0 |
| **核心检测项** | 读取 `$env:SystemRoot\System32\drivers\etc\hosts`，过滤注释行（`#`开头），检查是否存在非 `127.0.0.1`/`::1` 的激活条目；检测 127.0.0.1 映射到知名域名（被劫持指向本地）；文件编码异常检测（非 UTF-8/ANSI 或含 NULL 字节） |
| **降级策略** | 文件不存在 → OK（正常）；文件被锁不可读 → M9=UNKNOWN；编码损坏 → 标记但不中断 |

### 1.4 异常降级策略总表

| 降级类型 | 触发条件 | 处理方式 | 对后续影响 |
|----------|----------|----------|------------|
| **CMD_TIMEOUT** | 模块超时 | 标记模块状态=TIMEOUT，终止该模块所有子任务 | 下游依赖模块标记 SKIP |
| **CMD_FAIL** | Cmdlet/命令返回错误 | 回退方案尝试1次；仍失败 → 模块=UNKNOWN | 下游模块继续，但标记 UNCERTAIN |
| **PARTIAL** | 模块部分子项失败 | 模块=PASS（附带警告），失败子项记录到 JSON | 下游正常进行 |
| **SKIP** | 上游依赖失败/超时 | 模块不执行，直接标记 SKIP | 再次下游的模块也 SKIP |
| **CRITICAL** | WMI/CIM 服务完全不可用 | 全局降级，切换到纯文本解析模式 | 所有模块性能下降，超时翻倍 |

### 1.5 全诊断耗时预算

```
Phase 0: max(M1=800, M5=1500, M8=500, M9=300) = 1500ms
Phase 1: M2 = 600ms                                          → 累计 2100ms
Phase 2: max(M3=1200, M4=1000, M6=800) = 1200ms              → 累计 3300ms
Phase 3: M7 = 1000ms                                          → 累计 4300ms
─────────────────────────────────────────────────────────────
总理想耗时: ~4300ms < 5000ms ✓
```

**优化措施**：
- `Start-Job` 创建开销约 200ms/个，Phase 0 的 4 个 Job 实际耗时约 `max(子任务) + 200ms`
- 预加载公共模块（如 NetAdapter module import）在脚本启动时完成，不计入诊断耗时
- 诊断结果缓存到 `$global:DiagCache` hashtable，避免后续模块重复查询

## 2. 修复决策树

### 2.1 诊断结果 → 修复方案映射表

| 诊断发现 | 严重度 | 触发修复 | 自动/需确认 |
|----------|--------|----------|-------------|
| **M1: 适配器 Disabled** | 高 | `Enable-NetAdapter` | 自动（交互模式确认） |
| **M1: 适配器 Disconnected** | 信息 | 不触发修复，提示检查物理连接 | — |
| **M1: APIPA (169.254) + Status=Up** | 高 | DHCP 修复链：`ipconfig /release` → `/renew` → 重启 DHCP 服务 | 自动 |
| **M1: 驱动异常/缺失** | 高 | `Restart-NetAdapter` → 提示更新驱动（不自动安装） | 自动+提示 |
| **M2: 无默认网关** | 高 | 检查 DHCP → 若无 DHCP 则提示手动配置 | 提示（涉及用户决策） |
| **M2: DHCP 未启用** | 中 | 设置 `Set-NetIPInterface -Dhcp Enabled` | 自动 |
| **M2: IP 冲突** | 高 | `ipconfig /release` → `/renew` | 自动 |
| **M3: DNS 解析失败** | 高 | DNS 修复链：`/flushdns` → 重置 DNS 为自动获取 → 备选公共 DNS | 自动（公共 DNS 需确认） |
| **M4: 网关不可达** | 高 | ARP 清理 → 路由表修复 → 网关 ARP 重新解析 | 自动 |
| **M4: 路由表异常** | 中 | `route -f` + 重建默认路由（需确认，会清除静态路由） | 需确认 |
| **M5: 第三方驱动残留 (Start≠4)** | 中 | 禁用驱动（Set Start=4）→ 删除对应注册表项（需确认） | 需确认 |
| **M5: LSP 损坏/异常 DLL** | 高 | `netsh winsock reset`（会清除所有 LSP，需重启） | 需确认 |
| **M5: WFP 第三方过滤器** | 低 | 记录不自动修复，提示手动移除 | 提示 |
| **M5: TAP/Wintun 残留适配器** | 低 | 禁用/卸载虚拟适配器 | 需确认 |
| **M6: WinHTTP 代理残留** | 中 | `netsh winhttp reset proxy` | 自动 |
| **M6: IE/系统代理残留** | 中 | 注册表清除 ProxyEnable=0 | 自动 |
| **M6: VPN 残留路由** | 中 | `Remove-NetRoute` 删除异常路由条目 | 自动 |
| **M7: Winsock 损坏** | 高 | `netsh winsock reset` | 需确认（影响所有网络应用） |
| **M7: TCP-IP 栈损坏** | 高 | `netsh int ip reset` + `netsh int tcp reset` | 需确认 |
| **M8: 防火墙异常阻止** | 中 | `netsh advfirewall reset` → 恢复默认配置 | 需确认（会清除自定义规则） |
| **M8: 防火墙服务停止** | 高 | `Start-Service MpsSvc` | 自动 |
| **M9: Hosts 劫持** | 中 | 注释掉劫持行 → 备份原文件 → 清理 | 自动（保留备份） |

### 2.2 组合修复策略

当多个诊断结果同时命中时，按依赖顺序合并：

```
发现: M5(LSP损坏) + M3(DNS失败) + M7(Winsock损坏)
→ 合并为: netsh winsock reset (同时修复 LSP + Winsock)
→ 然后: DNS 修复链
→ 顺序不可逆
```

**合并规则**：
1. `netsh winsock reset` 覆盖 M5-LSP + M7-Winsock
2. `netsh int ip reset` 覆盖 M2-IP + M7-IP + 部分 M4 路由
3. `netsh int tcp reset` 覆盖 M7-TCP
4. `ipconfig /release + /renew` 覆盖 M1-APIPA + M2-IP冲突
5. ARP 清理在路由修复之前执行
6. 防火墙重置在 netsh 链之后执行

## 3. 修复分级

### 3.1 分级总览

| 级别 | 名称 | 命令集 | 风险 | 需重启 | 适用场景 |
|------|------|--------|------|--------|----------|
| **L1** | 轻量刷新 | `ipconfig /flushdns`, `arp -d *`, hosts 修复, `netsh winhttp reset proxy`, 注册表清除代理 | 极低 | 否 | DNS 缓存过期、hosts 劫持、代理残留 |
| **L2** | 中度量修复 | `ipconfig /release`, `ipconfig /renew`, `Restart-NetAdapter`, DNS 重置为自动, 删除异常路由 | 低 | 否（短暂断网 <5s） | DHCP 续租、IP 冲突、路由异常 |
| **L3** | 深度协议栈修复 | `netsh winsock reset`, `netsh int ip reset`, `netsh int ipv4 reset`, `netsh int ipv6 reset`, `netsh int tcp reset`, `netsh advfirewall reset` | 中 | **是** | Winsock 损坏、LSP 损坏、TCP-IP 栈损坏、防火墙异常 |
| **L4** | 终极重置 | L3 全部 + 禁用第三方残留驱动 + 卸载虚拟适配器 + `netcfg -d`（移除并重装所有网络适配器）+ `DISM /Online /Cleanup-Image /RestoreHealth`（可选） | 高 | **是**（强制） | L3 无法修复、协议栈深度损坏、多因素叠加 |

### 3.2 每级风险标签

#### L1 — 轻量刷新

| 风险维度 | 评估 |
|----------|------|
| **数据丢失风险** | 无 |
| **网络中断时长** | <2s（仅 flushdns，几乎无感） |
| **副作用** | ARP 清理后首包延迟略增（需重新 ARP 解析）；hosts 修复会备份原文件 |
| **回滚难度** | 低（hosts 有备份；代理可重设） |
| **适用自动化** | 完全可自动执行 |

#### L2 — 中度修复

| 风险维度 | 评估 |
|----------|------|
| **数据丢失风险** | 无 |
| **网络中断时长** | 3-10s（release/renew 期间断网） |
| **副作用** | 当前 TCP 连接全部断开；Restart-NetAdapter 短暂中断所有网络；静态 IP 用户需重新配置 |
| **回滚难度** | 中（需记住原静态 IP/DNS 配置） |
| **适用自动化** | DHCP 环境可自动；静态 IP 环境需确认 |

#### L3 — 深度协议栈修复

| 风险维度 | 评估 |
|----------|------|
| **数据丢失风险** | 低（不影响用户数据，但 VPN/虚拟网卡配置可能被清除） |
| **网络中断时长** | 需重启，中断 1-2 分钟 |
| **副作用** | `netsh winsock reset` 清除所有 LSP（含杀软/虚拟化软件注入的合法 LSP）；`netsh advfirewall reset` 清除所有自定义防火墙规则；`netsh int ip reset` 重置 TCP/IP 参数到出厂值 |
| **回滚难度** | **高**——LSP 不可逆恢复；防火墙需重新配置；建议执行前创建系统还原点 |
| **适用自动化** | 需用户明确确认 |

#### L4 — 终极重置

| 风险维度 | 评估 |
|----------|------|
| **数据丢失风险** | 中（`netcfg -d` 移除所有网络适配器并重装，静态 IP/VPN/虚拟交换机配置全部丢失） |
| **网络中断时长** | 需重启，中断 3-5 分钟 |
| **副作用** | 清除所有网络配置（含虚拟机网络、VPN 配置、Hyper-V 虚拟交换机）；禁用第三方驱动后该软件功能受限 |
| **回滚难度** | **极高**——建议执行前完整备份网络配置并创建系统还原点；`netcfg -d` 不可逆 |
| **适用自动化** | 需用户明确确认 + 最终二次确认 |

## 4. 全局状态机

### 4.1 状态转换图

```

                                   │

                          ┌────────────────┐

                     │  └──┬───┘   └────┬─────┘
                     │     │            │
                     │     ▼            ▼
```

### 4.2 状态定义

| 状态 | 说明 | 进入条件 | 合法出边 |
|------|------|----------|----------|
| `START` | 脚本入口 | — | CHECK_ADMIN |
| `CHECK_ADMIN` | 检测是否管理员运行 | START | ADMIN, NON_ADMIN |
| `ADMIN` | 管理员模式 | `[Security.Principal.WindowsPrincipal]::new(...).IsInRole([Security.Principal.WindowsBuiltInRole]::Administrator)` | MODE_SELECT |
| `NON_ADMIN` | 普通用户模式 | 非管理员 | MODE_SELECT (仅诊断可用) |
| `MODE_SELECT` | 模式选择 | ADMIN/NON_ADMIN | DIAGNOSE_ONLY, REPAIR_ONLY, FULL_AUTO |
| `DIAGNOSE_ONLY` | 仅诊断 | MODE_SELECT(所有模式共用入口) | DIAGNOSE |
| `REPAIR_ONLY` | 仅修复（需预加载诊断结果） | MODE_SELECT + `-RepairOnly` 参数 | DIAGNOSE(增量) |
| `FULL_AUTO` | 诊断+修复全自动 | MODE_SELECT(默认) | DIAGNOSE |
| `DIAGNOSE` | 执行诊断流水线 | DIAGNOSE_ONLY / FULL_AUTO / REPAIR_ONLY | ANALYZE (正常) / DIAGNOSE (重试) / ERROR |
| `ANALYZE` | 分析诊断结果，生成修复方案 | DIAGNOSE | REPAIR (有需修复项) / REPORT (无需修复) |
| `REPAIR` | 执行修复 | ANALYZE / FULL_AUTO | VERIFY (正常) / ANALYZE (重新分析) / ERROR |
| `VERIFY` | 快速回诊验证修复效果 | REPAIR | REPORT (通过) / ANALYZE (仍有问题，重新分析升级) / REPAIR (重试同级别) |
| `REPORT` | 生成并输出报告 | DIAGNOSE / VERIFY / ANALYZE | EXIT |
| `ERROR` | 全局错误状态 | 任意状态(异常) | EXIT / DIAGNOSE(重试) / CHECK_ADMIN(权限提升重试) |
| `EXIT` | 退出 | REPORT / ERROR | — |

### 4.3 异常转换规则

| 源状态 | 异常 | 目标状态 | 动作 |
|--------|------|----------|------|
| `CHECK_ADMIN` | 无法判断权限 | ERROR | 记录异常，询问是否继续(NON_ADMIN降级) |
| `DIAGNOSE` | 任意模块 CRITICAL 降级 | DIAGNOSE | 切换纯文本解析模式，重跑失败模块+下游 |
| `DIAGNOSE` | 全部模块返回 UNKNOWN | ERROR | 提示系统 WMI/CIM 服务异常 |
| `REPAIR` | L1/L2 命令执行失败 | REPAIR | 重试1次 → 升级到 L3 |
| `REPAIR` | L3 命令执行失败 | ERROR | 记录，推荐 L4 或手动干预 |
| `REPAIR` | L4 命令执行失败 | ERROR | 记录，提示联系系统管理员 |
| `VERIFY` | 问题未解决 | ANALYZE | 重新分析，升级修复级别 |
| `VERIFY` | 验证产生新问题 | ANALYZE | 标记回归，重新分析 |
| 任意 | 用户中断 (Ctrl+C) | EXIT | 记录中断状态，安全退出 |
| 任意 | 脚本内部异常(未捕获) | ERROR | 记录堆栈跟踪，优雅降级退出 |

### 4.4 控制流伪代码

```powershell
function Invoke-NetworkEmergencyKit {
    param([ValidateSet('Diagnose','Repair','Auto')][string]$Mode = 'Auto')

    $State = 'CHECK_ADMIN'
    $DiagResult = $null
    $RepairPlan = $null
    $RetryCount = @{ Diagnose = 0; Repair = 0 }

    while ($State -ne 'EXIT') {
        switch ($State) {
            'CHECK_ADMIN' {
                $global:IsAdmin = Test-AdminPrivilege
                $State = $global:IsAdmin ? 'ADMIN' : 'NON_ADMIN'
            }
            'ADMIN'      { $State = 'MODE_SELECT' }
            'NON_ADMIN'  {
                if ($Mode -in @('Repair','Auto')) {
                    Write-Warning "修复需要管理员权限，已降级为仅诊断模式"
                    $Mode = 'Diagnose'
                }
                $State = 'MODE_SELECT'
            }
            'MODE_SELECT' {
                $State = switch ($Mode) {
                    'Diagnose' { 'DIAGNOSE_ONLY' }
                    'Repair'   { 'REPAIR_ONLY' }
                    'Auto'     { 'FULL_AUTO' }
                }
            }
            'DIAGNOSE_ONLY' { $DiagResult = Invoke-DiagnosticPipeline; $State = 'ANALYZE' }
            'REPAIR_ONLY'   {
                $DiagResult = $DiagResult ?? (Invoke-DiagnosticPipeline -Targeted)
                $State = 'ANALYZE'
            }
            'FULL_AUTO'     { $DiagResult = Invoke-DiagnosticPipeline; $State = 'ANALYZE' }
            'ANALYZE' {
                $RepairPlan = New-RepairPlan -DiagResult $DiagResult -PreviousPlan $RepairPlan
                if ($RepairPlan.Steps.Count -eq 0) { $State = 'REPORT' }
                elseif ($Mode -eq 'Diagnose')      { $State = 'REPORT' }
                else                               { $State = 'REPAIR' }
            }
            'REPAIR' {
                try {
                    $RepairResult = Invoke-RepairPlan -Plan $RepairPlan
                    $State = 'VERIFY'
                } catch {
                    $RetryCount.Repair++
                    if ($RetryCount.Repair -le 1) { $State = 'REPAIR' }
                    else { $State = 'ERROR'; $LastError = $_ }
                }
            }
            'VERIFY' {
                $VerifyResult = Invoke-QuickVerify -DiagResult $DiagResult -RepairPlan $RepairPlan
                if ($VerifyResult.AllPassed) { $State = 'REPORT' }
                else {
                    $RepairPlan = $VerifyResult.UpgradedPlan
                    if ($RepairPlan.CanRetry) { $State = 'REPAIR' }
                    else { $State = 'ANALYZE' }
                }
            }
            'REPORT' { Write-Report -DiagResult $DiagResult -RepairPlan $RepairPlan; $State = 'EXIT' }
            'ERROR'  { Write-ErrorReport -Error $LastError; $State = 'EXIT' }
        }
    }
}
```

## 5. 错误处理矩阵

### 5.1 四级场景处理策略

#### 场景 A: 权限不足

| 触发条件 | 检测时机 | 表现 |
|----------|----------|------|
| 非管理员运行修复模式 | CHECK_ADMIN | 修复命令返回 Access Denied |

| 处理层 | 策略 |
|--------|------|
| **检测** | 脚本入口 `#Requires -RunAsAdministrator` 仅作为注释提示（不硬阻断），实际用 `Test-AdminPrivilege` 函数检测 |
| **诊断阶段** | 非管理员不影响诊断（Get-NetAdapter/Resolve-DnsName 等均可运行），仅 `netsh wfp show filters` 需要管理员——无权限则自动跳过 |
| **修复阶段** | 检测到非管理员：提示用户以管理员重新运行，提供两种选择：① 重新以管理员启动（`Start-Process powershell -Verb RunAs`）② 仅运行诊断并输出修复建议（用户手动执行命令） |
| **恢复** | 自动重新以管理员启动需用户确认 UAC 弹窗 |
| **报告** | 记录 `"elevation": "denied"`, 诊断结果标记 `"repairable": true, "repaired": false` |

#### 场景 B: 命令超时

| 触发条件 | 检测时机 | 表现 |
|----------|----------|------|
| 模块执行超过预设时间阈值 | DIAGNOSE / REPAIR | `Start-Job` 的 `Wait-Job -Timeout` 触发 |

| 处理层 | 策略 |
|--------|------|
| **诊断阶段** | 单模块超时 → 该模块标记 `TIMEOUT`，`Stop-Job` 终止；依赖它的下游模块标记 `SKIP`；不依赖的模块继续 |
| **修复阶段** | 修复命令超时 → 重试 1 次（超时值翻倍）；再次超时 → 该修复步骤标记 `TIMEOUT`，跳过继续执行下一步 |
| **nsLookup/DNS 类** | `Resolve-DnsName` 超时采用快速失败（`-DnsOnly -QuickTimeout`），不阻塞 |
| **恢复** | 超时的模块可在 `VERIFY` 阶段作为重点验证项回诊 |
| **报告** | 记录 `"status": "timeout"`, `"elapsed_ms": <实际耗时>`, `"threshold_ms": <阈值>` |

#### 场景 C: 部分失败 (Partial Failure)

| 触发条件 | 检测时机 | 表现 |
|----------|----------|------|
| 模块子任务部分成功、部分失败 | DIAGNOSE / REPAIR | M5 中注册表扫描成功但 WFP 检测失败; 修复中 5 步执行有 1 步失败 |

| 处理层 | 策略 |
|--------|------|
| **诊断阶段** | 模块状态标记 `PASS`（而非 FAIL），附加 `warnings` 数组记录失败子项详情 |
| **修复阶段** | 修复步骤间相互独立的不因某步失败而中止；存在依赖关系的失败则跳过下游步骤（标记 `SKIP_BY_DEPENDENCY`） |
| **分析阶段** | `ANALYZE` 对 PARTIAL 结果生成保守修复方案——仅触发确认可修复的项 |
| **验证阶段** | `VERIFY` 重点检查 `warnings` 中的失败子项对应的诊断指标 |
| **报告** | 记录 `"severity": "partial"`, 详细列出每个子项的 `pass/fail/timeout/skip` 状态 |

#### 场景 D: 完全失败 (Total Failure)

| 触发条件 | 检测时机 | 表现 |
|----------|----------|------|
| 全部诊断模块失败；全部修复步骤失败；WMI 服务不可用 | DIAGNOSE / REPAIR | 所有模块返回 UNKNOWN/TIMEOUT/FAIL |

| 处理层 | 策略 |
|--------|------|
| **检测** | `$DiagResult.Modules.Values.Status -notcontains 'PASS'` 判断 |
| **诊断全部失败** | 检查 WMI/CIM 服务状态；检查 PowerShell 执行策略；检查 .NET Framework 完整性；输出系统级故障报告 |
| **修复全部失败** | 检查系统文件完整性 (`sfc /scannow` 建议)；检查磁盘空间（`netsh` 等命令需要写入）；检查注册表访问权限 |
| **恢复** | 输出详细的手动修复指南（脱离脚本的操作步骤）；提示使用 Windows 自带"网络重置"功能 |
| **退出码** | 完整失败 → `exit 3`（区别于部分失败 `exit 2`, 成功 `exit 0`, 纯诊断通过 `exit 0`） |
| **报告** | 记录 `"severity": "critical"`, 附带系统环境快照（OS 版本、PowerShell 版本、.NET 版本、WMI 状态） |

### 5.2 错误码体系

| 退出码 | 含义 | 触发条件 |
|--------|------|----------|
| `0` | 成功 | 诊断通过（无问题）或修复全部成功 |
| `1` | 诊断发现问题（仅诊断模式） | 诊断发现需修复项，但因模式限制未修复 |
| `2` | 部分修复成功 | 修复执行完毕但部分步骤失败/超时 |
| `3` | 完全失败 | 诊断全部不可用或修复全部失败 |
| `4` | 权限不足 | 修复模式非管理员运行 |
| `5` | 用户中断 | Ctrl+C 或其他用户主动终止 |
| `6` | 参数错误 | 无效的命令行参数 |
