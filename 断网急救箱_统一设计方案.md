# 断网急救箱 (NetAid) — 统一设计方案

> Windows 网络诊断与修复工具 | PowerShell 5.1 零依赖 | Win10/11 全版本兼容 | 版本 1.0

---

## 一、架构设计

### 1.1 产品命名与编码规范

| 项目 | 取值 |
|------|------|
| 产品名 | 断网急救箱 (NetAid) |
| 内部代号 | NetRescueKit (仅代码仓库使用) |
| 启动器 | `NetAid.bat` (ASCII编码，无BOM，无中文) |
| 主脚本 | `NetAid.ps1` (UTF-8 with BOM) |
| 模块脚本 | `modules/*.ps1` (全部UTF-8 with BOM) |

编码原则见 CLAUDE.md 速查表：.bat纯ASCII由.ps1处理中文；.ps1必须UTF-8 BOM + `[Console]::OutputEncoding = UTF8`。

**依赖关系说明：**
- M02依赖M01的活跃适配器列表(InterfaceIndex)
- M03/M04/M10依赖M02的IP/网关/DNS配置信息
- M08需要串行执行(netsh命令互斥)，依赖前序所有模块完成
- M05/M07/M09无依赖，Phase 0即可启动

### 1.4 并行执行策略

```
Phase 0 ─ Start-Job 并行 (4个Job同时启动):
  Job_M01: 适配器检测       (~500ms)
  Job_M05: 防火墙状态       (~300ms)
  Job_M07: 第三方残留       (~1500ms ← 瓶颈，注册表扫描)
  Job_M09: Hosts劫持        (~200ms)
  → Phase 0 耗时 ≈ max(各Job) ≈ 1500ms

Phase 1 ─ 串行（依赖M01结果）:
  M02: IP配置与DHCP         (~600ms)
  → 累计耗时 ≈ 2100ms

Phase 2 ─ Start-Job 并行 (3个Job，依赖M02结果):
  Job_M03: DNS解析          (~1200ms ← 瓶颈，Resolve-DnsName超时)
  Job_M04: 路由表           (~500ms)
  Job_M10: IP冲突           (~300ms)
  → Phase 2 耗时 ≈ 1200ms，累计 ≈ 3300ms

Phase 3 ─ 串行（netsh互斥）:
  M08: Winsock/协议栈       (~1000ms)
  → 总诊断耗时 ≈ 4300ms < 5000ms ✓

Phase 4 ─ 修复（按L1→L2→L3→L4分级串行，同级内可并行）
Phase 5 ─ 验证（快速回诊 + 必要时全量回诊）
```

**关键约束：**
- 所有netsh子命令共享全局锁，修复阶段必须串行。诊断阶段的netsh show类命令为只读操作，可松散并行但M08单独隔离。
- Start-Job创建开销约200ms/个，已计入耗时预算。
- 预加载公共模块(NetAdapter模块导入等)在脚本启动时完成，不计入诊断耗时。
- 诊断结果缓存到`$global:DiagCache`，避免后续模块重复查询。

**超时控制：**
- 单模块最大超时5s，超时则`Stop-Job`并标记TIMEOUT
- 总诊断最大超时10s，超时则强制收集已有结果
- 依赖下游模块在上游超时/失败时自动标记SKIP

## 二、诊断模块详细设计

### 2.1 M01 -- 适配器状态检测

**检测逻辑：**

```
输入: 无
输出: { Adapters, ActiveAdapterCount, DriverIssues }
超时: 800ms

检查项1: 枚举网络适配器
  Get-NetAdapter | 获取所有适配器的 Name, Status, LinkSpeed, InterfaceDescription
  如果 Get-NetAdapter 失败 → WMI回退 Get-WmiObject Win32_NetworkAdapter

检查项2: 适配器异常状态
  Status='Disabled' → 标记"适配器已禁用"
  Status='Not Present' → 标记"硬件未检测到"
  Status='Disconnected' → 仅记录，不标记为故障(可能是物理网线未插)

检查项3: 驱动信息
  Get-NetAdapter | Get-NetAdapterAdvancedProperty
  辅助: Get-WmiObject Win32_NetworkAdapter | Where NetEnabled=$true
  检查 Installed 属性是否为 False

检查项4: PnP设备状态
  Get-PnpDevice -Class Net | Where Status -ne 'OK'
  Status='Error' → 标记驱动故障

检查项5: 系统事件日志 (最近24h)
  Get-WinEvent -LogName System -FilterXPath "*[System[EventID=10400]]" -MaxEvents 50
  EventID 10400 = NDIS重置/网卡问题
```

**故障判定：** 活跃网卡Status为Disabled/Not Present、或PnP设备状态为Error、或24h内存在NDIS错误事件 → 适配器/驱动故障。

**降级策略：**
- `Get-NetAdapter`失败 → 回退`Get-WmiObject Win32_NetworkAdapter`(较慢但兼容性更好)
- WMI完全不可用 → 标记M01=UNKNOWN，M02降级为`ipconfig /all`文本解析

---

### 2.2 M02 -- IP配置与DHCP

**检测逻辑：**

```
输入: $global:DiagCache.M01 (活跃适配器列表)
输出: { IPv4, IPv6, APIPA_Detected, DhcpServiceRunning, DhcpLeaseValid }
超时: 600ms
依赖: M01

检查项1: APIPA检测 (核心)
  $adapters = 从M01获取Status='Up'的适配器
  foreach ($adapter in $adapters):
    $ipInfo = Get-NetIPAddress -InterfaceIndex $adapter.InterfaceIndex -AddressFamily IPv4
    if ($ipInfo.IPAddress -match '^169\.254\.'):
      标记: APIPA故障
      记录: 适配器名称、IP、PrefixLength、接口索引
    ⚠️ 仅检测Status='Up'的适配器，已断开/禁用的适配器上的169.254不算故障

检查项2: DHCP租约状态
  Get-NetIPConfiguration -InterfaceIndex $idx | Select DHCP, IPv4Address
  如果 DHCP启用但无有效租约 + APIPA地址 → 确认为DHCP故障

检查项3: DHCP Client服务状态
  Get-Service Dhcp | 确认 Status='Running'
  如果停止 → 额外标记 DHCP_SERVICE_STOPPED

检查项4: IP配置完整性
  Get-NetIPAddress + Get-NetIPConfiguration
  检查: IPv4地址、子网掩码、默认网关、DNS服务器是否齐全
  IPv6独立评估(仅当有全局单播IPv6地址时才检查)
```

**故障判定：** 任一Up状态网卡持有169.254.x.x地址且DHCP已启用但无有效租约 → DHCP故障。DHCP Client服务未运行 → 附加故障。无默认网关 → 标记(由M04深入检测)。

**降级策略：**
- NetAdapter模块不可用 → 回退`ipconfig /all`文本解析
- 完全失败 → M03/M04/M10标记SKIP

---

### 2.3 M03 -- DNS解析

**检测逻辑：**

```
输入: M02的DNS服务器列表, 测试域名列表
输出: { DnsTests, DnsServersReachable, OverallVerdict }
超时: 1200ms
依赖: M02

测试域名列表: ["www.baidu.com", "www.microsoft.com", "dns.msftncsi.com"]

检查项1 (主): Resolve-DnsName 解析测试
  foreach ($domain in $testDomains):
    try:
      $result = Resolve-DnsName -Name $domain -Type A -ErrorAction Stop -DnsOnly -QuickTimeout
      记录: 解析成功/失败, 解析耗时
    catch:
      记录失败原因
  ⚠️ PS 5.1 Resolve-DnsName可用(非cmdlet,是函数)

检查项2 (辅): Ping ICMP可达性
  $pingTargets = @('223.5.5.5', '8.8.8.8', '114.114.114.114')
  foreach ($ip in $pingTargets):
    Test-Connection -ComputerName $ip -Count 1 -Quiet
  ⚠️ ICMP可能被防火墙阻止，Ping失败不直接判DNS故障

检查项3: DNS服务器配置与可达性
  Get-DnsClientServerAddress -AddressFamily IPv4
  Test-Connection到每个DNS服务器IP

检查项4: DNS Client服务状态
  Get-Service Dnscache | 确认 Running

判定矩阵:
  Resolve-DnsName全部成功 → PASS
  Resolve-DnsName失败但Ping成功 → 纯DNS故障(DNS_ONLY_FAILURE)
  Resolve-DnsName失败且Ping失败 → 网络层连通性问题(NETWORK_UNREACHABLE)
  同时检测到APIPA → DHPC故障(DHCP_ROOT_CAUSE)
```

**故障判定：** Resolve-DnsName在所有测试域名上均失败，或DNS服务器列表为空/全部不可达 → DNS故障。辅测试用于区分故障根因。

**降级策略：**
- `Resolve-DnsName`不可用 → 仅用`[System.Net.Dns]::GetHostEntry()`替代
- 完全失败 → M03=UNKNOWN

### 2.4 M04 -- 路由表检查

**检测逻辑：**

```
输入: M02的默认网关IP列表
输出: { DefaultGatewayExists, GatewayReachable, BlackholeRoutes, MetricConflicts }
超时: 800ms
依赖: M02

检查项1: 默认路由存在性
  $defaultRoutes = Get-NetRoute -DestinationPrefix '0.0.0.0/0'
  if ($defaultRoutes.Count -eq 0):
    标记: 无默认路由 → 严重故障

检查项2: 网关可达性 (ARP层)
  $gateways = $defaultRoutes.NextHop | Select -Unique
  foreach ($gw in $gateways):
    $arpEntry = Get-NetNeighbor -IPAddress $gw -ErrorAction SilentlyContinue
    if (-not $arpEntry -or $arpEntry.State -notin @('Reachable','Permanent')):
      标记: 网关ARP不可达

检查项3: 路由metric异常
  有线网卡metric应<无线网卡(通常自动metric)
  异常: 两个Up接口同时有默认路由且metric相同 → 潜在冲突

检查项4: 黑洞路由/异常持久路由
  Get-NetRoute -PolicyStore ActiveStore |
    Where { $_.NextHop -eq '0.0.0.0' -and $_.DestinationPrefix -ne '0.0.0.0/0' }
  存在 → 标记为可疑黑洞路由

检查项5: 默认路由关联接口状态
  foreach ($route in $defaultRoutes):
    if ((Get-NetAdapter -InterfaceIndex $route.InterfaceIndex).Status -ne 'Up'):
      标记: 默认路由关联接口非活动
```

**故障判定：** 无默认路由 → 严重故障。网关ARP不可达 → 中度故障。存在黑洞路由或多默认路由metric冲突 → 警告。

**降级策略：**
- `Test-Connection`超时 → 降级为仅ARP表+本地路由表判断
- 完全失败 → M04=UNKNOWN

---

### 2.5 M05 -- 防火墙状态

**检测逻辑：**

```
输入: 无
输出: { FirewallProfiles, ServiceRunning, AbnormalBlockRules, ThirdPartyTakeover }
超时: 500ms

检查项1: 防火墙配置文件状态
  Get-NetFirewallProfile -All
  检查: Domain/Private/Public三个配置文件的Enabled, DefaultInboundAction, DefaultOutboundAction
  如果 DefaultOutboundAction='Block' → 标记异常(可能导致断网)

检查项2: 出站阻断规则
  Get-NetFirewallRule -Direction Outbound -Action Block -Enabled True
  检查是否存在非默认出站阻止规则(DisplayName不匹配'Core|Windows|Microsoft')
  如果存在 → 标记

检查项3: MpsSvc服务状态
  Get-Service MpsSvc | 确认 Running
  如果停止 → 标记(可能是第三方防火墙卸载后残留)

检查项4: BFE(Base Filtering Engine)异常
  Get-Service BFE | 确认 Running
  检查是否有第三方过滤驱动占用了BFE
```

**故障判定：** DefaultOutboundAction=Block ➔ 严重故障。存在非默认出站阻止规则 ➔ 中度故障。MpsSvc未运行 ➔ 警告。

**降级策略：**
- Cmdlet不可用 → 回退`netsh advfirewall show allprofiles`文本解析
- 完全失败 → M05=UNKNOWN

---

### 2.6 M06 -- VPN/代理残留

**检测逻辑：**

```
输入: 无
输出: { WinHttpProxy, IEProxy, EnvProxy, VpnAdapters, VpnResidualRoutes }
超时: 800ms

检查项1: IE系统代理
  Get-ItemProperty "HKCU:\Software\Microsoft\Windows\CurrentVersion\Internet Settings"
  ProxyEnable=1 → 标记 + 记录ProxyServer
  AutoConfigURL非空 → 标记PAC自动配置

检查项2: WinHTTP代理
  netsh winhttp show proxy
  输出不包含"直接访问" → 标记WinHTTP代理已设置

检查项3: 环境变量代理
  $envVars = @('HTTP_PROXY','HTTPS_PROXY','NO_PROXY','http_proxy','https_proxy')
  检查用户级 + 系统级环境变量, 非空即标记

检查项4: VPN虚拟适配器
  Get-NetAdapter | Where InterfaceDescription -match 'TAP|Wintun|VPN|Virtual|PANGP|CheckPoint'
  记录虚拟适配器名称、状态
  Status='Disconnected'但有持久路由 → 残留风险

```

**故障判定：** 任一代理启用或VPN残留路由存在 → VPN/代理故障。

**降级策略：**
- 注册表读取失败 → 仅检查WinHTTP
- 完全失败 → M06=UNKNOWN

### 2.7 M07 -- 第三方安全软件残留

**检测逻辑：**

```
输入: 无
输出: { ActiveDrivers, GhostDrivers, LSP_Count, LSP_Corrupted, WFP_ForeignFilters, VirtualAdapters }
超时: 1500ms (4个子检测内并行)

子检测A: 注册表驱动扫描
  $servicesPath = "HKLM:\SYSTEM\CurrentControlSet\Services"
  匹配已知驱动黑名单(详见2.7.1)
  判定 Start 值:
    Start=0/1/2 + 驱动文件存在 → 活跃残留(高危)
    Start=0/1/2 + 驱动文件不存在 → 幽灵残留(中危)
    Start=3(Manual) → 潜在风险(低危)
    Start=4(Disabled) → 无害(仅记录)

子检测B: Winsock LSP目录
  路径: HKLM:\...\WinSock2\Parameters\Protocol_Catalog9\Catalog_Entries
  条目数<20 → 严重(LSP目录损坏)
  条目数>200 → 警告(可能有第三方LSP注入)
  遍历解析PackedCatalogItem中DLL路径, 校验文件存在性

子检测C: WFP过滤器
  netsh wfp show filters file=-
  解析输出, 匹配已知第三方Provider名称
  过滤器数量>800 → 警告(显著超出正常范围~350-500)

子检测D: TAP/Wintun虚拟适配器
  Get-NetAdapter | Where Description匹配虚拟适配器特征
  记录状态, 已断开的标记为残留

驱动文件存在性检查:
  if ($imagePath):
    $expandedPath = [Environment]::ExpandEnvironmentVariables($imagePath)
    $fileExists = Test-Path $expandedPath
```

**故障判定：** 存在Start≠4的已知残留驱动 → 第三方残留故障。LSP目录损坏 → 协议栈故障(触发的修复与M08重叠)。WFP过滤器异常多 → 警告。

**降级策略：**
- 注册表权限不足 → M07标记PARTIAL
- `netsh wfp`失败 → WFP子项标记SKIP
- 任一子项超时 → 子项标记TIMEOUT，其他子项继续

### 2.8 M08 -- Winsock/TCP-IP协议栈

**检测逻辑：**

```

  $winsockPath = "HKLM:\SYSTEM\CurrentControlSet\Services\WinSock2\Parameters\Protocol_Catalog9"
```

### 2.10 M10 -- IP冲突检测

**检测逻辑：**

```
输入: M02的本机IP列表
输出: { ArpConflicts, ConflictEvents_24h, GratuitousArpResponse }
超时: 500ms
依赖: M02

检查项1: ARP表IP-MAC冲突
  Get-NetNeighbor | 获取所有ARP条目
  对每个本机IP, 检查ARP表中是否有多个MAC地址声明同一IP
  如果有 → 标记IP冲突

检查项2: 系统事件日志
  Get-WinEvent -LogName System -FilterXPath "*[System[EventID=4199]]" -MaxEvents 10
  EventID 4199 = Windows IP地址冲突检测事件
  如果存在(24小时内) → 标记IP冲突

检查项3: Gratuitous ARP探测
  向本网段广播地址发送ARP探测包
  检查是否有其他设备响应本机IP
```

**故障判定：** ARP表存在IP-MAC不唯一、或24h内存在EventID 4199、或Gratuitous ARP有响应 → IP冲突故障。

**降级策略：**
- `Get-WinEvent`无权限 → 仅依赖ARP表和Gratuitous ARP
- 完全失败 → M10=UNKNOWN

## 三、修复模块详细设计

### 3.1 修复分级总览

| 级别 | 名称 | 典型命令 | 风险 | 需重启 | 网络中断 | 适用场景 |
|------|------|---------|------|--------|---------|----------|
| **L1** | 轻量刷新 | flushdns, hosts修复, 代理重置, ARP清理 | 极低 | 否 | <2s | DNS缓存/hosts劫持/代理残留 |
| **L2** | 网络层修复 | release/renew, 路由修复, DNS重置为自动 | 低 | 否 | 3-10s | DHCP续租/IP冲突/路由异常 |
| **L3** | 协议栈修复 | winsock/int ip/tcp reset, 防火墙重置 | 中 | **是** | 1-2min | Winsock损坏/LSP损坏/防火墙异常 |
| **L4** | 终极重置 | L3全部 + 驱动禁用 + netcfg -d + DISM | 高 | **是(强制)** | 3-5min | L3无法修复/深度损坏/多因素叠加 |

### 3.2 修复执行策略

**总原则：** 先执行无风险操作(L1→L2)，需要用户确认后执行深度操作(L3→L4)。同级别内无依赖则并行，netsh命令互斥则串行。

```
修复流水线:
  Phase 修复A (L1 - 无风险，无需确认)：
    并行: R09_HostsRepair + R06_ProxyClean
    串行: ARP缓存清理
    → 预计耗时 < 2s

  Phase 修复B (L2 - 低风险，网络短暂中断)：
    串行: R02_DhcpRenew → R03_DnsReset → R04_RouteFix
    (如果DhcpRenew成功则跳过后续部分步骤)
    → 预计耗时 5-15s (含ipconfig /renew等待)

  Phase 修复C (L3 - 中风险，需用户确认)：
    用户确认[Y/n] →
    串行: R08_WinsockReset(含TCP/IP) → R05_FirewallReset
    ⚠️ netsh命令互斥，严格串行
    → 预计耗时 10-20s

  Phase 修复D (L4 - 高风险，需用户二次确认)：
    用户二次确认[Y/n] →
    串行: R07_ThirdPartyClean → R01_AdapterRestart
    (会禁用第三方驱动、重启网卡)
    → 预计耗时 5-15s

  全部修复后统一提示:
    "有N项修复需要重启后生效，是否现在重启? [Y/n]"
```

**修复依赖约束：**
- R08_WinsockReset必须在所有netsh修复之前执行(它重置底层协议栈)
- R02_DhcpRenew不依赖R08(ipconfig /release + /renew走不同路径)
- R01_AdapterRestart会短暂断网，放在最后
- R07_ThirdPartyClean(禁用/删除残留驱动)可能需重启，放在L4
- 同级别内无依赖的修复可以并行

### 3.3 各修复模块详情

#### R01 -- 网卡驱动重启

| 项目 | 内容 |
|------|------|
| **触发条件** | M01检测到适配器驱动异常、Status异常 |
| **风险等级** | L4 (中-高，执行时网络短暂中断5-10s) |
| **修复步骤** | `Disable-NetAdapter -Name <name> -Confirm:$false` → `Enable-NetAdapter -Name <name>` |
| **额外步骤** | 如果PnP设备Error: `Restart-PnpDevice`；尝试从驱动存储重新安装: `Update-PnpDevice` |
| **验证方法** | 重新运行M01，确认Status恢复Up，无新事件日志错误 |
| **失败回退** | 提示从设备管理器更新驱动，或从主板厂商下载 |
| **前置确认** | 是(L4级别) |

#### R02 -- DHCP续租修复

| 项目 | 内容 |
|------|------|
| **触发条件** | M02检测到APIPA地址或DHCP租约异常 |
| **风险等级** | L2 (低，仅影响DHCP获取过程) |
| **修复步骤** | `ipconfig /release` → `ipconfig /renew` |
| **额外步骤** | 如果DHCP Client服务未运行: `Start-Service Dhcp` |
| **验证方法** | 重新运行M02，确认IP不在169.254范围且获取到有效网关 |
| **失败回退** | 设置静态IP引导用户手动配置 |
| **前置确认** | 否 |

#### R03 -- DNS重置修复

| 项目 | 内容 |
|------|------|
| **触发条件** | M03检测到DNS解析失败 |
| **风险等级** | L2 (低) |
| **修复步骤** | `ipconfig /flushdns` → `ipconfig /registerdns` |
| **额外步骤** | `Clear-DnsClientCache`；重置DNS为自动获取: `Set-DnsClientServerAddress -InterfaceIndex <idx> -ResetServerAddresses` |
| **验证方法** | 重新运行M03，Resolve-DnsName解析成功 |
| **失败回退** | 设置公共DNS (114.114.114.114 / 223.5.5.5)，需用户确认 |
| **前置确认** | 否(公共DNS回退时才确认) |

#### R04 -- 路由表修复

| 项目 | 内容 |
|------|------|
| **触发条件** | M04检测到路由表异常 |
| **风险等级** | L2 (低-中) |
| **修复步骤** | 删除指向Disconnected接口的持久路由 → 重建默认路由 → 清除metric异常 |
| **具体命令** | `Remove-NetRoute` 删除冗余/可疑路由；`New-NetRoute` 重建默认路由(如果确认网关IP) |
| **验证方法** | 重新运行M04，确认默认网关存在且ARP可达 |
| **失败回退** | 提示手动配置路由表 |
| **前置确认** | 否(仅删除路由时可逆) |

#### R05 -- 防火墙重置

| 项目 | 内容 |
|------|------|
| **触发条件** | M05检测到防火墙异常 |
| **风险等级** | L3 (中，清空所有自定义规则) |
| **修复步骤** | `netsh advfirewall reset` (恢复出厂默认) |
| **额外步骤** | 确保MpsSvc和BFE服务正在运行 |
| **验证方法** | 重新运行M05，确认配置文件Enabled=True, DefaultOutboundAction=Allow |
| **失败回退** | 备份规则: `netsh advfirewall export "backup.wfw"` |
| **前置确认** | 是(L3级别) |

#### R06 -- 代理/VPN清理

| 项目 | 内容 |
|------|------|
| **触发条件** | M06检测到代理/VPN残留 |
| **风险等级** | L1 (极低) |
| **修复步骤** | |
| 步骤1 | IE代理关闭: `Set-ItemProperty "HKCU:\...\Internet Settings" -Name ProxyEnable -Value 0` |
| 步骤2 | WinHTTP重置: `netsh winhttp reset proxy` |

| 步骤4 | VPN残留路由清除: `Remove-NetRoute` 清除M06检测出的无效路由 |
| **验证方法** | 重新运行M06，确认无代理启用 + 无VPN残留路由 |
| **失败回退** | 手动清除注册表 |
| **前置确认** | 否 |

#### R07 -- 第三方残留清理

| 项目 | 内容 |
|------|------|
| **触发条件** | M07检测到第三方残留驱动/过滤器 |
| **风险等级** | L4 (高，可能需重启，影响安全软件功能) |
| **修复步骤** | |
| 步骤1 | 活跃残留(start≠4): 设置Start=4(禁用) → 提示重启后生效 |
| 步骤2 | 幽灵残留(文件不存在): 删除注册表Service键 |
| 步骤3 | LSP损坏: 触发netsh winsock reset (与R08合并) |
| 步骤4 | WFP过滤器: 记录不自动修复(需手动处理或netsh advfirewall reset覆盖) |
| **验证方法** | 重新运行M07，确认驱动Start变为4，注册表键已清理 |
| **失败回退** | 注册表备份 + 系统还原点 |
| **前置确认** | 是(L4级别，需二次确认) |

#### R08 -- Winsock/TCP-IP协议栈重置

| 项目 | 内容 |
|------|------|
| **触发条件** | M08检测到协议栈异常，或用户选择"一键重置网络栈" |
| **风险等级** | L3 (中，部分VPN/安全软件需重装) |
| **前置提示** | "警告: 部分VPN和第三方防火墙软件在重置后需要重新安装。" |
| **修复步骤** | 严格按此顺序(netsh互斥): |
| 1 | `netsh winsock reset` |
| 2 | `netsh int ip reset` |
| 3 | `netsh int ipv4 reset` |
| 4 | `netsh int ipv6 reset` |
| 5 | `netsh int tcp reset` |
| 6 | `netsh winhttp reset proxy` |
| 7 | `netsh advfirewall reset` |
| **验证方法** | 重新运行M08，检查Winsock注册表项完整性 |
| **失败回退** | `sfc /scannow`建议；提示创建系统还原点 |
| **前置确认** | 是(L3级别) |

#### R09 -- hosts文件修复

| 项目 | 内容 |
|------|------|
| **触发条件** | M09检测到hosts劫持或损坏 |
| **风险等级** | L1 (极低) |
| **修复步骤** | 备份当前hosts到backups/ → 注释掉劫持行 → 清理异常条目 |
| **默认内容** | 不强制设为默认(保留用户自定义条目)，仅清理被标记为劫持的行 |
| **验证方法** | 重新运行M09，确认关键域名无劫持条目 |
| **失败回退** | 从.bak备份恢复 |
| **前置确认** | 否 |

#### R10 -- IP冲突修复

| 项目 | 内容 |
|------|------|
| **触发条件** | M10检测到IP冲突 |
| **风险等级** | L2 (低) |
| **修复步骤** | `ipconfig /release` → `ipconfig /renew` (重新获取新IP) |
| **额外步骤** | `arp -d *` 清理本地ARP缓存 |
| **验证方法** | 重新运行M10，确认无冲突事件 |
| **失败回退** | 提示用户检查局域网内是否有静态IP冲突设备 |
| **前置确认** | 否 |

### 3.4 修复决策合并规则

当多个诊断结果同时命中时，按以下规则合并修复方案：

| 规则 | 合并策略 |
|------|---------|
| **netsh winsock reset** | 同时覆盖 M07-LSP + M08-Winsock |
| **netsh int ip reset** | 同时覆盖 M02-IP + M08-IP栈 + 部分M04-路由 |
| **netsh int tcp reset** | 覆盖 M08-TCP栈 |
| **ipconfig /release + /renew** | 同时覆盖 M02-APIPA + M10-IP冲突 |
| **netsh advfirewall reset** | 同时覆盖 M05-防火墙 + M07-部分WFP过滤器 |
| **ARP清理** | 在路由修复之前执行 |
| **防火墙重置** | 在netsh链之后执行 |

## 五、执行流程

### 5.1 诊断流水线

**Phase 0 (无依赖模块，4 Job并行):**

| Job | 模块 | 超时 | 产出 |
|-----|------|------|------|
| Job1 | M01_Adapter | 800ms | 适配器列表、状态 |
| Job2 | M05_Firewall | 500ms | 防火墙配置、规则 |
| Job3 | M07_ThirdParty | 1500ms | 残留驱动、LSP、WFP |
| Job4 | M09_Hosts | 300ms | hosts劫持条目 |

**Phase 1 (依赖M01，串行):**

| 模块 | 超时 | 依赖 | 产出 |
|------|------|------|------|
| M02_IP_DHCP | 600ms | M01 (活跃适配器列表) | IP/DHCP配置、APIPA |

**Phase 2 (依赖M02，3 Job并行):**

| Job | 模块 | 超时 | 依赖 | 产出 |
|-----|------|------|------|------|
| Job5 | M03_DNS | 1200ms | M02 (DNS服务器) | DNS解析结果 |
| Job6 | M04_Route | 800ms | M02 (网关IP) | 路由表分析 |
| Job7 | M10_IPConflict | 500ms | M02 (本机IP) | IP冲突检测 |

**Phase 3 (netsh互斥，串行):**

| 模块 | 超时 | 产出 |
|------|------|------|
| M08_Winsock | 1000ms | 协议栈健康状态 |

**耗时预算：**

```
Phase 0: max(800, 500, 1500, 300) = 1500ms
Phase 1: 600ms                         → 累计 2100ms
Phase 2: max(1200, 800, 500) = 1200ms  → 累计 3300ms
Phase 3: 1000ms                        → 累计 4300ms
────────────────────────────────────────
总理想耗时: ~4300ms < 5000ms ✓
```

**超时与降级矩阵：**

| 降级类型 | 触发条件 | 处理方式 | 对后续影响 |
|----------|----------|----------|-----------|
| CMD_TIMEOUT | 模块超时(单模块5s) | 标记模块=TIMEOUT, Stop-Job | 下游依赖模块标记SKIP |
| CMD_FAIL | Cmdlet返回错误 | 回退方案尝试1次;仍失败→UNKNOWN | 下游继续，标记UNCERTAIN |
| PARTIAL | 部分子项失败 | 模块=PASS(附warnings), 失败子项记录 | 下游正常进行 |
| SKIP | 上游依赖失败/超时 | 模块不执行，标记SKIP | 再下游也SKIP |
| CRITICAL | WMI/CIM完全不可用 | 全局降级，切换到纯文本解析 | 所有模块性能下降，超时翻倍 |

```
M03: NETWORK_UNREACHABLE → 检查M04/M02结果综合判断
M04: 无默认路由 ────────→ L2 ──→ 自动+提示(R04_RouteFix)
```

#### 5.3.2 状态定义

| 状态 | 说明 | 进入条件 | 合法出边 |
|------|------|----------|----------|
| START | 脚本入口 | -- | CHECK_ADMIN |
| CHECK_ADMIN | 检测管理员权限 | START | ADMIN, NON_ADMIN |
| ADMIN | 管理员模式 | IsInRole(Administrator) | MODE_SELECT |
| NON_ADMIN | 普通用户模式 | 非管理员 | MODE_SELECT(仅诊断可用) |
| MODE_SELECT | 模式选择 | ADMIN/NON_ADMIN | DIAGNOSE_ONLY, REPAIR_ONLY, FULL_AUTO |
| DIAGNOSE_ONLY | 仅诊断(含-Check) | MODE_SELECT | DIAGNOSE |
| REPAIR_ONLY | 仅修复(需预加载诊断) | MODE_SELECT + -Fix参数 | DIAGNOSE(增量) |
| FULL_AUTO | 诊断+修复全自动 | MODE_SELECT(-Auto) | DIAGNOSE |
| DIAGNOSE | 执行诊断流水线 | DIAGNOSE_ONLY/FULL_AUTO/REPAIR_ONLY | ANALYZE(正常)/DIAGNOSE(重试)/ERROR |
| ANALYZE | 分析结果, 生成修复方案 | DIAGNOSE | REPAIR(有需修复项)/REPORT(无需修复) |
| REPAIR | 执行修复(L1→L2自动, L3→L4确认) | ANALYZE/FULL_AUTO | VERIFY(正常)/ANALYZE(重新分析)/ERROR |
| VERIFY | 快速回诊验证 | REPAIR | REPORT(通过)/ANALYZE(仍有问题, 升级级别)/REPAIR(重试同级) |
| REPORT | 生成输出报告 | DIAGNOSE/VERIFY/ANALYZE | EXIT |
| ERROR | 全局错误 | 任意状态(异常) | EXIT/DIAGNOSE(重试)/CHECK_ADMIN(提权重试) |
| EXIT | 退出 | REPORT/ERROR | -- |

#### 5.3.3 异常转换规则

| 源状态 | 异常 | 目标状态 | 动作 |
|--------|------|----------|------|
| CHECK_ADMIN | 无法判断权限 | ERROR | 记录异常，询问是否NON_ADMIN降级继续 |
| DIAGNOSE | 任意模块CRITICAL降级 | DIAGNOSE | 切换纯文本解析模式，重跑失败模块+下游 |
| DIAGNOSE | 全部模块返回UNKNOWN | ERROR | 提示WMI/CIM服务异常 |
| REPAIR | L1/L2命令失败 | REPAIR | 重试1次 → 仍失败则跳过继续 |
| REPAIR | L3命令失败 | ERROR | 记录，推荐L4或手动干预 |
| REPAIR | L4命令失败 | ERROR | 记录，提示联系系统管理员 |
| VERIFY | 问题未解决 | ANALYZE | 重新分析，升级修复级别 |
| VERIFY | 验证产生新问题 | ANALYZE | 标记回归，重新分析 |
| 任意 | 用户中断(Ctrl+C) | EXIT | 记录中断状态，安全退出 |
| 任意 | 脚本内部异常 | ERROR | 记录堆栈跟踪，优雅降级退出 |

#### 5.3.4 VERIFY验证策略

```
快速回诊(默认):
  对每个修复过的模块，重新运行对应的诊断
  全部通过 → VERIFY_PASS → REPORT
  仍有失败 → 升级修复级别 → 重新ANALYZE

全量回诊(升级后):
  VERIFY失败一次后触发
  跳过已验证通过的模块，仅重诊失败模块+新增疑似关联模块
  如果仍然失败 → 标记为需手动干预 → REPORT附加手动修复指南
```

### 5.4 完整交互流程

```
启动 → 解析参数 → 权限检查 → 系统环境检测 → 模式路由
  │
  ├─ 无参数(交互模式):
  │   主菜单 → 用户选择 → 诊断 → 报告 → 修复确认 → 执行 → 验证 → 完成
  │
  ├─ -Auto:
  │   全诊断 → L1+L2自动修复 → L3+L4逐项确认 → 验证 → 报告 → 退出(含退出码)
  │
  ├─ -Check:
  │   全诊断 → 控制台报告 + JSONL日志 → 退出(exit 0/1)
  │
  ├─ -Fix <模块>:
  │   定向诊断 → 分析 → 修复 → 验证 → 退出
  │
  └─ -Silent:
      同其他参数但控制台无彩色输出, 仅StdOut输出JSON
```

### 5.5 修复分级决策流程

```
诊断完成 → ANALYZE
  │
  ├─ 无问题 → REPORT → EXIT(0)
  │
  └─ 有问题 → 生成修复计划
       │
       ├─ L1项(hosts/proxy/flushdns): 直接自动执行
       │
       ├─ L2项(dhcp/dns/route/ipconflict): 自动执行
       │   失败 → 重试1次 → 仍失败 → 升级到L3
       │
       ├─ L3项(winsock/firewall): 提示用户确认
       │   "以下操作需要重置网络协议栈/防火墙，执行后部分VPN/安全软件需重新安装。"
       │   确认 → 执行 → 验证
       │   拒绝 → 跳过，REPORT标记"用户跳过"
       │
       └─ L4项(thirdparty/adapter): 提示用户二次确认
           "以下操作将禁用第三方驱动/重启网卡，可能导致网络短暂中断或安全软件功能受限。"
           确认 → 执行 → 标记reboot_required
           拒绝 → 跳过，REPORT标记"用户跳过"
```

## 六、错误处理与日志

### 6.1 四级错误处理

#### 场景A: 权限不足

| 维度 | 内容 |
|------|------|
| 触发条件 | 非管理员运行修复模式 |
| 检测时机 | CHECK_ADMIN |
| 诊断阶段 | 非管理员不影响诊断，仅`netsh wfp show filters`需管理员(无权限则自动跳过) |
| 修复阶段 | 提示用户以管理员重新运行，提供两种选择：①自动提权重启(`Start-Process powershell -Verb RunAs`) ②仅运行诊断并输出修复建议(用户手动执行) |
| 退出码 | 4 |

#### 场景B: 命令超时

| 维度 | 内容 |
|------|------|
| 触发条件 | 模块超过预设时间阈值 |
| 诊断阶段 | 单模块超时→标记TIMEOUT, Stop-Job; 依赖下游标记SKIP; 不依赖的继续 |
| 修复阶段 | 修复命令超时→重试1次(超时值翻倍); 再次超时→标记TIMEOUT, 跳过继续下一步 |
| 恢复 | 超时模块在VERIFY阶段作为重点验证项回诊 |

#### 场景C: 部分失败

| 维度 | 内容 |
|------|------|
| 触发条件 | 模块子任务部分成功、部分失败 |
| 诊断阶段 | 模块标记PASS(附warnings数组记录失败子项) |
| 修复阶段 | 独立步骤间不因某步失败而中止; 有依赖关系的失败则跳过下游(SKIP_BY_DEPENDENCY) |
| 分析阶段 | ANALYZE对PARTIAL结果生成保守修复方案(仅确认可修复项) |
| 退出码 | 2 |

#### 场景D: 完全失败

| 维度 | 内容 |
|------|------|
| 触发条件 | 全部诊断模块失败或全部修复步骤失败 |
| 检测 | 检查WMI/CIM服务状态、PowerShell执行策略、.NET Framework完整性 |
| 恢复 | 输出详细手动修复指南；提示使用Windows自带"网络重置"功能 |
| 退出码 | 3 |
| 报告 | 附带系统环境快照(OS版本、PS版本、.NET版本、WMI状态) |

### 6.2 日志系统

#### 6.2.1 日志文件

- 格式: JSON Lines (.jsonl)，每行一条独立JSON
- 编码: UTF-8 (无BOM)

### 7.2 BAT启动器 (NetAid.bat)

```bat
@echo off
setlocal enabledelayedexpansion

:: ============================================================
::  NetAid.bat — 断网急救箱启动器
::  职责: 权限检测 / UAC提权 / PowerShell策略绕过 / 参数透传
::  编码: ASCII (纯英文, 无中文, 无BOM)
:: ============================================================

set "SCRIPT_DIR=%~dp0"
set "PS1_SCRIPT=%SCRIPT_DIR%NetAid.ps1"

:: 0. 检查PS脚本是否存在
if not exist "%PS1_SCRIPT%" (
    echo [ERROR] NetAid.ps1 not found: %PS1_SCRIPT%
    echo Please ensure NetAid.ps1 is in the same directory as NetAid.bat
    pause
    exit /b 1
)

:: 1. 检测管理员权限
net session >nul 2>&1
if %errorlevel% equ 0 (set "IS_ADMIN=1") else (set "IS_ADMIN=0")

:: 2. 收集传入参数
set "ARGS="
:parse_args
if "%~1"=="" goto args_done
set "ARGS=%ARGS% %1"
shift
goto parse_args
:args_done

:: 3. 非管理员 → VBScript ShellExecute UAC提权
if "%IS_ADMIN%"=="0" (
    echo [INFO] Requesting administrator privileges...
    set "VBS_FILE=%TEMP%\NetAid_Elevate.vbs"
    (
        echo Set UAC = CreateObject^("Shell.Application"^)
        echo UAC.ShellExecute "cmd.exe", "/c """"%SCRIPT_DIR%NetAid.bat""" %ARGS%", "", "runas", 1
    ) > "%VBS_FILE%"
    cscript //nologo "%VBS_FILE%" >nul 2>&1
    del "%VBS_FILE%" >nul 2>&1
    exit /b %errorlevel%
)

:: 4. 已是管理员: 设置标题并调用PS
title NetAid - Network First Aid

powershell.exe -NoProfile -ExecutionPolicy Bypass -Command ^
    "[Console]::OutputEncoding = [System.Text.Encoding]::UTF8; ^
     $OutputEncoding = [System.Text.Encoding]::UTF8; ^
     & '%PS1_SCRIPT%' %ARGS%; ^
     exit $LASTEXITCODE"

set "EXIT_CODE=%errorlevel%"
if "%EXIT_CODE%" neq "0" (
    echo.
    echo [NetAid] Exit code: %EXIT_CODE%
)

exit /b %EXIT_CODE%
```

**架构决策说明：**

| 方案 | 方式 | 优缺点 |
|------|------|--------|
| **VBScript ShellExecute("runas")** (采用) | 纯Windows原生，零依赖，UAC对话框最干净 | 不依赖PS已可用 |
| PowerShell Start-Process -Verb RunAs (备选) | PS自举 | 需PS可用，参数含空格转义复杂 |
| runas命令 (不采用) | 需要目标用户明文密码 | 不适合UAC提权场景 |

```powershell
    [Parameter(ParameterSetName = 'Check')]
    [switch] $Check,

    [Parameter(ParameterSetName = 'Fix')]

    [Parameter(ParameterSetName = 'Check')]
    [Parameter(ParameterSetName = 'Fix')]
    [Parameter(ParameterSetName = 'Auto')]

    [Parameter(ParameterSetName = 'Fix')]
    [Parameter(ParameterSetName = 'Auto')]

    [string] $Output = 'Console',

    [Parameter(ParameterSetName = 'Check')]

}

# 加载公共库
. "$PSScriptRoot\lib\Utils.ps1"
. "$PSScriptRoot\lib\Logger.ps1"
. "$PSScriptRoot\lib\UI.ps1"
. "$PSScriptRoot\lib\Parallel.ps1"

# ... (后续: 参数路由, 状态机驱动, 诊断引擎, 修复引擎等)
```
