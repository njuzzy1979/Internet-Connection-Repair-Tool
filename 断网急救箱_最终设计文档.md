## 一、概述

### 1.1 产品定义

| 项目 | 取值 |
|------|------|
| 产品名 | **断网急救箱 (NetAid)** |
| 目标用户 | Windows 10 1809+ / Windows 11 全版本普通用户、IT运维人员 |
| 核心能力 | 覆盖10类网络故障的一键诊断与分级自动修复 |
| 运行环境 | Windows PowerShell 5.1 (系统内置)，零第三方依赖 |
| 修复权限 | 诊断可非管理员运行；修复需管理员权限(自动UAC提权) |

### 1.2 设计原则

1. **PS 5.1 零依赖**：所有功能在 Windows 10/11 自带 PowerShell 5.1 上即可运行，不引入 Npcap/WinPcap、第三方模块或外部可执行文件。
2. **API 务实校准**：所有 cmdlet/参数使用前已核验 PS 5.1 兼容性(审查发现 `-DnsOnly`/`-QuickTimeout` 等不存在于 5.1 的参数已被替代方案替换)。
3. **分级修复(安全第一)**：L1(轻量零风险)→L2(网络层，需检测远程会话)→L3(协议栈，需确认)→L4(驱动级，需二次确认)，L3/L4 前强制提示系统还原点创建。
4. **不可逆操作备份**：所有修改注册表/重置防火墙/修改 hosts 等操作前强制备份到 `backups/` 目录。
5. **工具自清理**：不写注册表，不安装服务，删除整个目录即卸载。日志默认写入 `%TEMP%\NetAid\`。

### 1.3 适用范围与限制

| 维度 | 说明 |
|------|------|
| 操作系统 | Windows 10 1809+ / Windows 11 全版本 |
| PowerShell | Windows PowerShell 5.1 (非 PowerShell 7) |
| IPv4/IPv6 | 双栈支持：诊断覆盖 IPv4 与 IPv6(独立评估，互不影响) |
| 远程会话 | RDP/SSH/WinRM 场景下自动降级执行策略(跳过网卡断开操作) |
| 企业环境 | 检测 ConstrainedLanguage/AppLocker/WDAC，提供手动诊断路径 |
| 物理连接问题 | 不属于诊断范围(Status=Disconnected 仅提示，不触发修复) |

### 2.4 并行执行策略 (含数据传递修正)

**核心修正(审查 P0-3)**：`Start-Job` 创建独立 PowerShell 进程，父作用域变量不可见。改为 **Phase 间屏障模式**：每个 Phase 完成后通过 `Receive-Job` 收集结果并写入 `$global:DiagCache`，下一 Phase 启动时通过 `-ArgumentList` 传入所需数据切片。

```
Phase 0 ─ Start-Job 并行 (4个Job同时启动):
  Job_M01: 适配器检测       (~500ms)
  Job_M05: 防火墙状态       (~300ms)
  Job_M07: 第三方残留       (~1500ms ← 瓶颈，注册表扫描)
  Job_M09: Hosts劫持        (~200ms)
  → Wait-Job 所有Job → Receive-Job 收集 → 写入 $global:DiagCache
  → Phase 0 耗时 ≈ max(各Job) + 收集开销 ≈ 1500ms

Phase 1 ─ 串行（必须等Phase 0收集完成后执行）:
  M02: IP配置与DHCP         (~600ms)
  → 写入 $global:DiagCache → 累计耗时 ≈ 2100ms

Phase 2 ─ Start-Job 并行 (3个Job，通过-ArgumentList传入M02数据):
  Job_M03: DNS解析          (~3000ms ← 修正：Start-Job超时控制替代DnsOnly参数)
  Job_M04: 路由表           (~500ms)
  Job_M10: IP冲突           (~300ms)
  → Wait-Job → Receive-Job 收集
  → Phase 2 耗时 ≈ 3000ms，累计 ≈ 5100ms

Phase 3 ─ 串行（netsh互斥，可直接访问$global:DiagCache）:
  M08: Winsock/协议栈       (~1000ms)
  → 总诊断耗时 ≈ 6100ms
```

**耗时预算修正(审查 P0-1 关联)：** 因 PS 5.1 无法使用 `-QuickTimeout` 参数，M03 DNS 解析需通过 `Start-Job` + `Wait-Job -Timeout` 实现超时控制，单次 `[System.Net.Dns]::GetHostEntry()` 默认超时约 5-15s，Job 级超时设为 3s。总诊断耗时上限从原本 5s 修正为约 6s。

**关键约束：**
- 所有 `netsh` 子命令共享全局锁，修复阶段必须串行。诊断阶段的 `netsh show` 类命令为只读操作，可松散并行但 M08 单独隔离。
- `Start-Job` 创建开销约 200ms/个，已计入耗时预算。
- 预加载公共模块(NetAdapter 模块导入等)在脚本启动时完成，不计入诊断耗时。
- **子 Job 不直接写日志文件**：日志事件通过 `Receive-Job` 收集后由主线程统一写入 JSONL，避免并发写入竞态(P2-2)。

**超时控制：**
- 单模块最大超时 5s，超时则 `Stop-Job` 并标记 TIMEOUT
- 总诊断最大超时 12s，超时则强制收集已有结果
- WFP 子检测独立超时(15s，Start-Job 内)，超时仅标记 WFP=SKIP
- 依赖下游模块在上游超时/失败时自动标记 SKIP/SKIP_BY_DEPENDENCY

## 三、诊断模块详细设计

### 3.1 M01 — 适配器状态检测

**检测逻辑：**

```
输入: 无
输出: { Adapters, ActiveAdapterCount, DriverIssues, WlanSvcRunning, WiFiSignalQuality, AirplaneMode, NcsiProbeFailed }
超时: 800ms

检查项1: 枚举网络适配器
  Get-NetAdapter | 获取所有适配器的 Name, Status, LinkSpeed, InterfaceDescription
  如果 Get-NetAdapter 失败 → WMI回退 Get-WmiObject Win32_NetworkAdapter

检查项2: 适配器异常状态
  Status='Disabled' → 标记"适配器已禁用"
  Status='Not Present' → 标记"硬件未检测到"
  Status='Disconnected' → 仅记录，不标记为故障(可能是物理网线未插)

检查项3: 驱动信息
  Get-NetAdapter | Get-NetAdapterAdvancedProperty
  辅助: Get-WmiObject Win32_NetworkAdapter | Where NetEnabled=$true
  检查 Installed 属性是否为 False

检查项4: PnP设备状态
  Get-PnpDevice -Class Net | Where Status -ne 'OK'
  Status='Error' → 标记驱动故障

检查项5: 系统事件日志 (最近24h) (修正 P3-2: 使用 -FilterHashtable)
  Get-WinEvent -FilterHashtable @{LogName='System'; Id=10400; StartTime=(Get-Date).AddHours(-24)} -MaxEvents 50
  EventID 10400 = NDIS重置/网卡问题

检查项6: WLAN AutoConfig 服务 (新增 P2-4)
  Get-Service WlanSvc | 确认状态
  如果未运行且存在无线适配器 → 标记WLAN_SERVICE_STOPPED

检查项7: Wi-Fi信号 (新增 P2-4)
  netsh wlan show interfaces | 解析SSID/Signal/Band
  仅当无线适配器Status='Up'时执行
  信号<30% → 标记WIFI_SIGNAL_WEAK

检查项8: 飞行模式 (新增 P2-4)
  注册表: HKLM\SYSTEM\CurrentControlSet\Control\RadioManagement
  检查是否有EnableRadio=0的项 → 标记AIRPLANE_MODE_ON

检查项9: NCSI探测状态 (新增 P3-4)
  Get-ItemProperty "HKLM:\SYSTEM\CurrentControlSet\Services\NlaSvc\Parameters\Internet"
  检查 EnableActiveProbing 是否为 1
  检查 ActiveWebProbeHost 是否可 PING: Test-Connection dns.msftncsi.com -Count 1 -Quiet
```

**故障判定：** 活跃网卡 Status 为 Disabled/Not Present、或 PnP 设备状态为 Error、或 24h 内存在 NDIS 错误事件 → 适配器/驱动故障。WlanSvc 未运行与无线适配器同时存在 → Wi-Fi 故障。NCSI 探针不可达且实际有网络 → NCSI 故障。

**降级策略：**
- `Get-NetAdapter` 失败 → 回退 `Get-WmiObject Win32_NetworkAdapter`(较慢但兼容性更好)
- WMI 完全不可用 → 标记 M01=UNKNOWN，M02 降级为 `ipconfig /all` 文本解析

### 3.2 M02 — IP配置与DHCP

**检测逻辑：**

```
输入: $global:DiagCache.M01 (活跃适配器列表)
输出: { IPv4, IPv6, APIPA_Detected, DhcpServiceRunning, DhcpLeaseValid, MtuValues, MtuAbnormal, IPv6_Enabled, IPv6_GlobalAddresses }
超时: 600ms
依赖: M01

检查项1: APIPA检测 (核心)
  $adapters = 从M01获取Status='Up'的适配器
  foreach ($adapter in $adapters):
    $ipInfo = Get-NetIPAddress -InterfaceIndex $adapter.InterfaceIndex -AddressFamily IPv4
    if ($ipInfo.IPAddress -match '^169\.254\.'):
      标记: APIPA故障
      记录: 适配器名称、IP、PrefixLength、接口索引
    ⚠️ 仅检测Status='Up'的适配器，已断开/禁用的适配器上的169.254不算故障

检查项2: DHCP租约状态
  Get-NetIPConfiguration -InterfaceIndex $idx | Select DHCP, IPv4Address
  如果 DHCP启用但无有效租约 + APIPA地址 → 确认为DHCP故障

检查项3: DHCP Client服务状态
  Get-Service Dhcp | 确认 Status='Running'
  如果停止 → 额外标记 DHCP_SERVICE_STOPPED

检查项4: IP配置完整性
  Get-NetIPAddress + Get-NetIPConfiguration
  检查: IPv4地址、子网掩码、默认网关、DNS服务器是否齐全
  IPv6独立评估(仅当有全局单播IPv6地址时才检查) (新增 P2-1)

检查项5: IPv6地址采集 (新增 P2-1)
  Get-NetIPAddress -AddressFamily IPv6 | Where PrefixOrigin -ne 'WellKnown'
  记录全局单播地址列表
  检查IPv6是否启用: Get-NetIPv6Protocol | Select Enabled

检查项6: MTU检查 (新增 P2-5)
  Get-NetIPInterface | Select ifIndex, InterfaceAlias, NlMtu
  异常判定: Mtu < 1280 (IPv6最低要求) 或 Mtu > 1500 (标准以太网，非Jumbo Frame场景) → 标记MTU_ABNORMAL
```

**故障判定：** 任一 Up 状态网卡持有 169.254.x.x 地址且 DHCP 已启用但无有效租约 → DHCP 故障。DHCP Client 服务未运行 → 附加故障。MTU 异常 → 警告。

**降级策略：**
- NetAdapter 模块不可用 → 回退 `ipconfig /all` 文本解析
- 完全失败 → M03/M04/M10 标记 SKIP

### 3.3 M03 — DNS解析

**修正(审查 P0-1)：** PS 5.1 的 `Resolve-DnsName` **不支持** `-DnsOnly` 和 `-QuickTimeout` 参数。M03 改为通过 `Start-Job` + `Wait-Job -Timeout` 实现超时控制和并发解析。优先使用 `[System.Net.Dns]::GetHostEntry()` 进行解析（更可控），`Resolve-DnsName` 作为补充。

**修正(审查 P2-1)：** 增加 IPv6 AAAA 记录测试。

**修正(审查 P3-3)：** 扩展测试域名列表，减少单域名劫持误判。

**检测逻辑：**

```
输入: M02的DNS服务器列表, 测试域名列表
输出: { DnsTests_A, DnsTests_AAAA, PingTests, DnsServersReachable, OverallVerdict }
超时: 3000ms (内部每个域名解析Job有2s独立超时)
依赖: M02

测试域名列表(多样化):
  IPv4: ["www.baidu.com", "cloudflare-dns.com", "dns.google", "www.apple.com"]
  IPv6: ["ipv6.baidu.com", "ipv6.google.com"] (仅当IPv6启用时)
  NCSI: ["dns.msftncsi.com"]

检查项1 (主): Start-Job超时控制的DNS解析
  # 实现示意 (不依赖 -DnsOnly/-QuickTimeout):
  $job = Start-Job -ArgumentList $domain -ScriptBlock {
      param($d)
      try {
          # 优先使用 .NET DNS (可设超时，但默认值较长)
          $result = [System.Net.Dns]::GetHostEntry($d)
          return @{Success=$true; IPs=$result.AddressList; Method='DotNet'}
      } catch {
          # 降级为 Resolve-DnsName (PS 5.1 内置函数)
          try {
              $r = Resolve-DnsName -Name $d -Type A -ErrorAction Stop
              return @{Success=$true; IPs=$r.IPAddress; Method='PS5'}
          } catch {
              return @{Success=$false; Error=$_.Exception.Message}
          }
      }
  }
  $job | Wait-Job -Timeout 2   # 每个域名最多等2秒
  $result = $job | Receive-Job
  $job | Remove-Job -Force

检查项2 (辅): Ping ICMP可达性
  $pingTargets = @('223.5.5.5', '8.8.8.8', '114.114.114.114')
  foreach ($ip in $pingTargets):
    Test-Connection -ComputerName $ip -Count 1 -Quiet
  ⚠️ ICMP可能被防火墙阻止，Ping失败不直接判DNS故障

检查项3: DNS服务器配置与可达性
  Get-DnsClientServerAddress -AddressFamily IPv4
  Get-DnsClientServerAddress -AddressFamily IPv6 (新增 P2-1)
  Test-Connection到每个DNS服务器IP

检查项4: DNS Client服务状态
  Get-Service Dnscache | 确认 Running

检查项5: IPv6 AAAA记录测试 (新增 P2-1)
  仅当M02检测到IPv6全局地址时执行
  使用与检查项1相同的Start-Job模式，-Type AAAA
  IPv6测试结果独立判定(失败不影响IPv4 PASS)

判定矩阵:
  Resolve-DnsName全部IPv4域名成功 → PASS
  部分成功/部分失败 → DEGRADED(记录失败域名)
  全部失败但Ping成功 → DNS_ONLY_FAILURE → FAIL
  全部失败且Ping失败 → NETWORK_UNREACHABLE(网络层故障)
  同时检测到APIPA → DHCP_ROOT_CAUSE(根因是DHCP)
```

**故障判定：** 所有测试域名解析失败，或 DNS 服务器列表为空/全部不可达 → DNS 故障。辅测试用于区分故障根因。

**降级策略：**
- `[System.Net.Dns]::GetHostEntry()` 和 `Resolve-DnsName` 均不可用 → M03=UNKNOWN
- Job 创建失败 → 串行执行替换并行方案

### 3.4 M04 — 路由表检查

**检测逻辑：**

```
输入: M02的默认网关IP列表
输出: { DefaultGatewayExists_v4, DefaultGatewayExists_v6, GatewayReachable, GatewayReachabilityRanking, BlackholeRoutes, MetricConflicts, PmtuProbeResult }
超时: 800ms
依赖: M02

检查项1: 默认路由存在性 (IPv4 + IPv6)
  $defaultRoutes_v4 = Get-NetRoute -DestinationPrefix '0.0.0.0/0'
  $defaultRoutes_v6 = Get-NetRoute -DestinationPrefix '::/0' (新增 P2-1)
  if ($defaultRoutes_v4.Count -eq 0): 标记: 无IPv4默认路由 → 严重故障
  if ($defaultRoutes_v6.Count -eq 0 -and IPv6已启用): 标记: 无IPv6默认路由 → 警告

检查项2: 网关可达性 (ARP层) + 多网关排序 (新增 P2-6)
  $gateways = $defaultRoutes.NextHop | Select -Unique
  foreach ($gw in $gateways):
    $arpEntry = Get-NetNeighbor -IPAddress $gw -ErrorAction SilentlyContinue
    if (-not $arpEntry -or $arpEntry.State -notin @('Reachable','Permanent')):
      标记: 网关ARP不可达
  对所有网关按可达性排序: 可达+有线>可达+无线>不可达

检查项3: 路由metric异常
  有线网卡metric应<无线网卡(通常自动metric)
  异常: 两个Up接口同时有默认路由且metric相同 → 潜在冲突

检查项4: 黑洞路由/异常持久路由
  Get-NetRoute -PolicyStore ActiveStore |
    Where { $_.NextHop -eq '0.0.0.0' -and $_.DestinationPrefix -ne '0.0.0.0/0' }
  存在 → 标记为可疑黑洞路由

检查项5: 默认路由关联接口状态
  foreach ($route in $defaultRoutes):
    if ((Get-NetAdapter -InterfaceIndex $route.InterfaceIndex).Status -ne 'Up'):
      标记: 默认路由关联接口非活动

检查项6: PMTU探测 (新增 P2-5)
  向网关发送带DF标志的ICMP包，探测路径MTU
  Test-Connection -MtuSize -MtuSizeBuffer 1500
  如果指定大小的包无法到达 → 标记MTU_PATH_ISSUE
```

**故障判定：** 无默认路由 → 严重故障。网关 ARP 不可达 → 中度故障。存在黑洞路由或多默认路由 metric 冲突 → 警告。PMTU 问题 → 警告。

**降级策略：**
- `Test-Connection` 超时 → 降级为仅 ARP 表+本地路由表判断
- 完全失败 → M04=UNKNOWN

#### 3.7.1 通信模块

每个模块通过 `Invoke-Module` 统一入口调用：

```powershell
function Invoke-Module {
    param(
        [string]$ModuleName,       # CLI短名: "dns"/"adapter"/...
        [string]$Mode,             # "diagnose" | "repair" | "verify"
        [hashtable]$Context = @{}  # 传入的依赖数据
    )
    $internalName = $Script:ModuleMap[$ModuleName]
    $scriptPath  = Join-Path $PSScriptRoot "modules\$internalName.ps1"

    # 修复阶段权限检查
    if ($Mode -eq 'repair' -and -not $Script:IsAdmin) {
        return @{ Verdict = 'ERROR'; Reason = 'ADMIN_REQUIRED' }
    }

    # dot-source 加载模块脚本并调用标准入口函数
    . $scriptPath
    return Invoke-DiagnoseModule -Context $Context
    # 或 Invoke-RepairModule -Context $Context
}
```

### 3.8 M08 — Winsock/TCP-IP协议栈

**检测逻辑：**

```

检查项1: Winsock目录完整性
  $winsockPath = "HKLM:\SYSTEM\CurrentControlSet\Services\WinSock2\Parameters\Protocol_Catalog9"

  如果被恶意关闭 → 标记

检查项5: 关键网络服务
  $services = @("Dhcp", "Dnscache", "LanmanWorkstation", "NlaSvc", "netprofm", "WlanSvc")
  foreach ($svc in $services):
    if Status -ne 'Running' 且 $svc 为关键服务 → 标记
```

**故障判定：** 任一关键注册表项缺失或参数异常 → 协议栈故障。

**降级策略：**
- netsh 命令全部失败 → M08=UNKNOWN(通常意味着协议栈严重损坏)
- 注册表读取失败 → 标记 PARTIAL

### 3.9 M09 — hosts文件劫持

**检测逻辑：**

```
输入: 无
输出: { HijackEntries, FileEncoding_OK, EncodingDetails, TotalActiveEntries, Permissions_OK }
超时: 300ms

检查项1: hosts文件存在性
  $hostsPath = "$env:SystemRoot\System32\drivers\etc\hosts"
  if (-not (Test-Path $hostsPath)): 标记文件不存在

检查项2: 劫持条目检测
  $criticalDomains = @("microsoft.com","windows.com","live.com","office.com",
                        "baidu.com","qq.com","taobao.com","jd.com")
  foreach ($line in Get-Content $hostsPath):
    跳过注释行(^\s*#)和空行
    对每个关键域名检查是否被映射到非0.0.0.0/127.0.0.1/::1的IP
    如果是 → 标记劫持

检查项3: 文件权限
  Get-Acl $hostsPath → 检查当前用户是否有读取权限

检查项4: 编码异常 (具体化 P3-5)
  $bytes = [System.IO.File]::ReadAllBytes($hostsPath)
  -- 检测1: BOM检测
     if ($bytes[0..1] -eq @(0xFF, 0xFE)): 标记 UTF16_LE_BOM_DETECTED
     if ($bytes[0..2] -eq @(0xEF, 0xBB, 0xBF)): 标记 UTF8_BOM_DETECTED
  -- 检测2: NULL字节检测 (表示UTF-16但未检测到BOM)
     if (($bytes | Where { $_ -eq 0 }).Count -gt 10): 标记 NULL_BYTES_DETECTED
  -- 检测3: C0控制字符检测 (排除 \r\n\t)
     允许的C0字符: 0x09(Tab), 0x0A(LF), 0x0D(CR), 0x20+
     其他C0(0x00-0x1F,排除上述三个) → 标记 ABNORMAL_CONTROL_CHARS
```

**故障判定：** 关键域名被映射到非本机 IP → hosts 劫持故障。文件不存在或权限异常 → 故障。

**降级策略：**
- 文件不存在 → PASS(正常状态，非故障)
- 文件被锁不可读 → M09=UNKNOWN
- 编码损坏 → 标记但不中断

### 3.10 M10 — IP冲突检测

**修正(审查 P1-1)：** Gratuitous ARP 发包探测在纯 PS 5.1 零依赖环境下不可实现（无 raw socket cmdlet）。降级为可选/移除，仅依赖检查项 1+2。

**修正(审查 P3-2)：** 将 `Get-WinEvent -FilterXPath` 替换为 `-FilterHashtable`，解决非英文 Windows 的 XPath 本地化兼容问题。

**检测逻辑：**

```
输入: M02的本机IP列表
输出: { ArpConflicts, ConflictEvents_24h }
超时: 500ms
依赖: M02

检查项1: ARP表IP-MAC冲突
  Get-NetNeighbor | 获取所有ARP条目
  对每个本机IP, 检查ARP表中是否有多个MAC地址声明同一IP
  如果有 → 标记IP冲突

检查项2: 系统事件日志 (修正: 使用 -FilterHashtable)
  Get-WinEvent -FilterHashtable @{LogName='System'; Id=4199; StartTime=(Get-Date).AddHours(-24)} -MaxEvents 10
  EventID 4199 = Windows IP地址冲突检测事件
  如果存在(24小时内) → 标记IP冲突
  ⚠️ 如果Get-WinEvent无权限 → 仅依赖检查项1

检查项3: Gratuitous ARP探测 — 降级为可选 (P1-1)
  此检测在纯PS 5.1零依赖环境下不可实现(需raw socket发ARP包)
  如需完整ARP探测能力，需引入Npcap/WinPcap或bundled arping.exe
  默认标记为NOT_AVAILABLE，不影响IP冲突结果判定
```

**故障判定：** ARP 表存在 IP-MAC 不唯一、或 24h 内存在 EventID 4199 → IP 冲突故障。

**降级策略：**
- `Get-WinEvent` 无权限 → 仅依赖检查项1
- 检查项1也可正常执行(最基础检测)
- 完全失败 → M10=UNKNOWN

## 四、修复模块详细设计

### 4.1 修复分级总览 (含审查修正)

| 级别 | 名称 | 典型命令 | 风险 | 需重启 | 网络中断 | 适用场景 |
|------|------|---------|------|--------|---------|----------|
| **L1** | 轻量刷新 | flushdns, hosts修复, 代理重置, ARP清理(用netsh interface ip delete arpcache) | 极低 | 否 | 0s | DNS缓存/hosts劫持/代理残留 |
| **L2** | 网络层修复 | renew(release跳过远程会话), 路由修复(网关可达性排序), DNS重置为自动 | 低 | 否 | 3-10s | DHCP续租/IP冲突/路由异常 |
| **L3** | 协议栈修复 | winsock/int ip reset(合并去冗余), 防火墙重置(备份完整) | 中 | **是** | 1-2min | Winsock损坏/LSP损坏/防火墙异常 |
| **L4a** | 驱动级修复 | 禁用第三方驱动+重启网卡 | 中-高 | 可能需要 | 5-10s | 单网卡驱动异常 |
| **L4b** | 终极重置 | L3全部 + netcfg -d + DISM /RestoreHealth | **极高** | **是(强制)** | **30min+** | L4a无法修复/深度损坏/多因素叠加 |

**L4 拆分说明(审查 P0-2)：**
- `DISM /Online /Cleanup-Image /RestoreHealth` 单次执行通常需 15-45 分钟(机械硬盘可超 1 小时)。
- `netcfg -d` 会移除并重装所有网络适配器(OHyper-V 虚拟交换机、静态IP、VLAN配置、VM网络全部丢失)。
- L4a 为相对安全的驱动级操作，L4b 为不可逆的终极手段。
- L4b 执行前必须：①完整网络配置快照(`netsh dump` + `Get-NetAdapter` JSON 导出) ②检查 DISM 磁盘空间(需≥5GB) ③检查 Windows Update 服务状态 ④显示明确时间来预估和不可逆损失警告。

### 4.2 修复执行策略

**总原则：** 先执行无风险操作(L1→L2)，需用户确认后执行深度操作(L3→L4)。同级别内无依赖则并行，netsh 命令互斥则串行。远程会话检测 + 备份机制贯穿全程。

```
修复流水线:
  Phase 修复A (L1 - 无风险，无需确认)：
    → 检测是否远程会话: 标记REMOTE_SESSION_FLAG
    并行: R09_HostsRepair + R06_ProxyClean
    串行: ARP缓存清理 (使用 netsh interface ip delete arpcache)
    → 预计耗时 < 2s

  Phase 修复B (L2 - 低风险，网络短暂中断)：
    → 远程会话检测: 若是远程会话，跳过 ipconfig /release
    [系统还原点提示] → 用户可跳过
    串行: R02_DhcpRenew(跳过release若远程) → R03_DnsReset → R04_RouteFix
    → 预计耗时 5-15s

  Phase 修复C (L3 - 中风险，需用户确认)：
    [强制系统还原点提示]
    "警告: netsh winsock reset 将清除所有 LSP(含杀毒软件实时网络防护、虚拟化软件网络组件、游戏反作弊驱动等)。"
    "受影响的软件可能需要重新安装其网络组件。"
    执行前检测受影响的LSP条目列表并展示
    用户确认[Y/n] →
    串行: R08_WinsockReset(仅保留 netsh int ip reset <logfile>, 移除ipv4/ipv6冗余) → R05_FirewallReset
    ⚠️ netsh命令互斥，严格串行
    → 预计耗时 10-20s

  Phase 修复D (L4 - 高风险，需用户二次确认)：
    [强制系统还原点提示]
    L4a:
      串行: R07_ThirdPartyClean → R01_AdapterRestart
      (会备份注册表+禁用第三方驱动+重启网卡)
      → 预计耗时 5-15s
    L4b (仅在L4a无效后):
      [强制执行完整网络配置快照 + DISM前检查]
      [显示时间预估: 约15-45分钟 + 不可逆警告]
      "此操作将移除并重装所有网络适配器。Hyper-V虚拟交换机、静态IP配置、VLAN配置将全部丢失。VM网络将全部中断。"
      用户二次确认 → 执行 netcfg -d → DISM /RestoreHealth → 重启

  全部修复后统一提示:
    "有N项修复需要重启后生效，是否现在重启? [Y/n]"
```

**修复依赖约束(含审查修正)：**
- R08_WinsockReset 必须在所有 netsh 修复之前执行(它重置底层协议栈)
- R02_DhcpRenew 不依赖 R08(ipconfig /release + /renew 走不同路径)
- **远程会话检测**：若为 RDP/SSH/WinRM 会话，R02 跳过 `ipconfig /release`(P1-2)
- R01_AdapterRestart 会短暂断网，放在 L4a 最后
- R07_ThirdPartyClean(禁用/删除残留驱动)需 reg export 备份，放在 L4a
- 同级别内无依赖的修复可以并行

|------|------|
| **触发条件** | M10检测到IP冲突 |
| **风险等级** | L2 (低) |
| **修复步骤** | `ipconfig /release` → `ipconfig /renew` (重新获取新IP)。远程会话跳过release。 |
| **ARP清理** | `netsh interface ip delete arpcache` (比 arp -d * 更可靠，兼容性更好, P2-8) |
| **验证方法** | 重新运行M10，确认无冲突事件 |
| **失败回退** | 提示用户检查局域网内是否有静态IP冲突设备 |
| **前置确认** | 否 |

### 4.4 修复决策合并规则

当多个诊断结果同时命中时，按以下规则合并修复方案：

| 规则 | 合并策略 |
|------|---------|
| **netsh winsock reset** | 同时覆盖 M07-LSP + M08-Winsock |
| **netsh int ip reset** | 同时覆盖 M02-IP + M08-IP栈 + 部分M04-路由 |
| **netsh int tcp reset** | 覆盖 M08-TCP栈 |
| **ipconfig /release + /renew** | 同时覆盖 M02-APIPA + M10-IP冲突 (远程会话跳过release) |
| **netsh advfirewall reset** | 同时覆盖 M05-防火墙 + M07-部分WFP过滤器 |
| **ARP清理 (netsh interface ip delete arpcache)** | 在路由修复之前执行 (P2-8) |
| **防火墙重置** | 在 netsh 链之后执行 |

## 六、执行流程

### 6.1 诊断流水线 (Phase分阶段 + 屏障收集)

```powershell
# ============================================================
# 诊断引擎示意 (主线程统一协调)
# ============================================================

# --- Phase 0: 无依赖模块 (4 Job并行) ---
$phase0_Jobs = @()
$phase0_Jobs += Start-Job -Name "M01" -ScriptBlock ${function:Invoke-M01_Diagnose}
$phase0_Jobs += Start-Job -Name "M05" -ScriptBlock ${function:Invoke-M05_Diagnose}
$phase0_Jobs += Start-Job -Name "M07" -ScriptBlock ${function:Invoke-M07_Diagnose}
$phase0_Jobs += Start-Job -Name "M09" -ScriptBlock ${function:Invoke-M09_Diagnose}

# 等待全部完成
$phase0_Jobs | Wait-Job | Out-Null

# 收集结果到主线程并写入 $global:DiagCache (P0-3修正)
foreach ($job in $phase0_Jobs) {
    $result = $job | Receive-Job
    $Script:DiagCache[$job.Name] = $result
    $result.LogEvents | ForEach-Object { Write-JsonlLog $_ }  # 主线程写入日志 (P2-2修正)
    $job | Remove-Job -Force
}

# --- Phase 1: 依赖M01，串行执行 ---
$m02_result = Invoke-M02_Diagnose -AdapterList $Script:DiagCache.M01.Adapters
$Script:DiagCache.M02 = $m02_result.Diagnosis
$m02_result.LogEvents | ForEach-Object { Write-JsonlLog $_ }

# --- Phase 2: 依赖M02，3 Job并行 ---
# 通过 -ArgumentList 传入所需数据切片 (P0-3修正)
$phase2_Jobs = @()
$phase2_Jobs += Start-Job -Name "M03" -ArgumentList $Script:DiagCache.M02 -ScriptBlock {
    param($M02Data)
    Invoke-M03_Diagnose -Context $M02Data
}
$phase2_Jobs += Start-Job -Name "M04" -ArgumentList $Script:DiagCache.M02 -ScriptBlock {
    param($M02Data)
    Invoke-M04_Diagnose -Context $M02Data
}
$phase2_Jobs += Start-Job -Name "M10" -ArgumentList $Script:DiagCache.M02 -ScriptBlock {
    param($M02Data)
    Invoke-M10_Diagnose -Context $M02Data
}

$phase2_Jobs | Wait-Job | Out-Null
foreach ($job in $phase2_Jobs) {
    $result = $job | Receive-Job
    $Script:DiagCache[$job.Name] = $result.Diagnosis
    $result.LogEvents | ForEach-Object { Write-JsonlLog $_ }
    $job | Remove-Job -Force
}

# --- Phase 3: netsh互斥，串行 (可直接访问$global:DiagCache) ---
$m08_result = Invoke-M08_Diagnose -DiagCache $Script:DiagCache
$Script:DiagCache.M08 = $m08_result.Diagnosis
$m08_result.LogEvents | ForEach-Object { Write-JsonlLog $_ }
```

**耗时预算(已修正)：**

```
Phase 0: max(800, 500, 1500, 300) + Job收集 ≈ 1700ms
Phase 1: 600ms                              → 累计 2300ms
Phase 2: max(3000, 800, 500) + Job收集 ≈ 3200ms → 累计 5500ms
Phase 3: 1000ms                             → 累计 6500ms
────────────────────────────────────────────
预期总耗时: ~6500ms (上限约10s)
```

### 6.2 超时与降级矩阵

| 降级类型 | 触发条件 | 处理方式 | 对后续影响 |
|----------|----------|----------|-----------|
| CMD_TIMEOUT | 模块超时(单模块5s) | 标记模块=TIMEOUT, Stop-Job | 下游依赖标记SKIP |
| SKIP_BY_DEPENDENCY | 上游步骤最终失败 | 模块=TIMEOUT, 记录跳过的依赖链 (新增P2-7) | 再下游也标记 |
| CMD_FAIL | Cmdlet返回错误 | 回退方案尝试1次;仍失败→UNKNOWN | 下游继续，标记UNCERTAIN |
| PARTIAL | 部分子项失败 | 模块=PASS(附warnings), 失败子项记录 | 下游正常进行 |
| SKIP | 上游依赖失败/超时 | 模块不执行，标记SKIP | 再下游也SKIP |
| CRITICAL | WMI/CIM完全不可用 | 全局降级，切换到纯文本解析 | 所有模块性能下降，超时翻倍 |

```
                        │       EXIT       │
                        └──────────────────┘
```

#### 6.3.2 状态定义 (含审查修正)

| 状态 | 说明 | 进入条件 | 合法出边 |
|------|------|----------|----------|
| START | 脚本入口 | -- | CHECK_ADMIN |
| CHECK_ADMIN | 检测管理员权限 | START | ADMIN, NON_ADMIN |
| CHECK_PS_ENV | PS运行环境检测(新增 P3-6) | ADMIN | 检查`$ExecutionContext.SessionState.LanguageMode`；若ConstrainedLanguage+AppLocker/WDAC→输出手动诊断路径并退出 |
| NON_ADMIN | 普通用户模式 | 非管理员 | MODE_SELECT(仅诊断可用) |
| MODE_SELECT | 模式选择 | CHECK_PS_ENV/NON_ADMIN | DIAGNOSE_ONLY, REPAIR_ONLY, FULL_AUTO |
| DIAGNOSE_ONLY | 仅诊断(含-Check) | MODE_SELECT | DIAGNOSE |
| REPAIR_ONLY | 仅修复(需预加载诊断) | MODE_SELECT + -Fix参数 | DIAGNOSE(增量) |
| FULL_AUTO | 诊断+修复全自动 | MODE_SELECT(-Auto) | DIAGNOSE |
| DIAGNOSE | 执行诊断流水线 | DIAGNOSE_ONLY/FULL_AUTO/REPAIR_ONLY | ANALYZE(正常)/DIAGNOSE(重试)/ERROR |
| ANALYZE | 分析结果, 生成修复方案 | DIAGNOSE | REPAIR(有需修复项)/REPORT(无需修复) |
| REPAIR | 执行修复(L1→L2自动, L3→L4确认) | ANALYZE/FULL_AUTO | VERIFY(正常)/ANALYZE(重新分析)/ERROR |
| VERIFY | 渐进式验证(快速→全量) | REPAIR | REPORT(通过)/ANALYZE(未通过, 升级)/REPAIR(重试同级) |
| REPORT | 生成输出报告 | DIAGNOSE/VERIFY/ANALYZE | EXIT |
| ERROR | 全局错误 | 任意状态(异常) | EXIT/DIAGNOSE(重试)/CHECK_ADMIN(提权重试) |
| EXIT | 退出 | REPORT/ERROR | -- |

#### 6.3.3 异常转换规则 (含审查修正)

| 源状态 | 异常 | 目标状态 | 动作 |
|--------|------|----------|------|
| CHECK_ADMIN | 无法判断权限 | ERROR | 记录异常，询问是否NON_ADMIN降级继续 |
| CHECK_PS_ENV | ConstrainedLanguage模式 | EXIT | 输出明确手动诊断指南路径（P3-6） |
| DIAGNOSE | 任意模块CRITICAL降级 | DIAGNOSE | 切换纯文本解析模式，重跑失败模块+下游 |
| DIAGNOSE | 全部模块返回UNKNOWN | ERROR | 提示WMI/CIM服务异常 |
| REPAIR | L1/L2命令失败 | REPAIR | 重试1次 → 仍失败则标记SKIP_BY_DEPENDENCY，下游依赖步骤跳过（P2-7） |
| REPAIR | L3命令失败 | ERROR | 记录，推荐L4或手动干预 |
| REPAIR | L4a命令失败 | ERROR | 记录，推荐L4b或手动干预 |
| REPAIR | L4b命令失败 | ERROR | 记录，输出手动修复指南+系统环境快照 |
| VERIFY | 问题未解决 | ANALYZE | 快速验证失败→全量回诊→重新分析，升级修复级别 |
| VERIFY | 验证产生新问题 | ANALYZE | 标记回归，重新分析 |
| 任意 | 用户中断(Ctrl+C) | EXIT | 记录中断状态，安全退出 |
| 任意 | 脚本内部异常 | ERROR | 记录堆栈跟踪，优雅降级退出 |

#### 6.3.4 VERIFY渐进式验证策略

```
第一轮: 快速验证
  对每个修复过的模块，重新运行对应的诊断
  全部通过 → VERIFY_PASS → REPORT
  仍有失败 → 记录失败模块，标记"升级修复级别"

第二轮: 全量回诊 (第一轮失败后触发)
  跳过已验证通过的模块
  仅重诊失败模块 + 新增疑似关联模块(跨模块级联影响)
  如果通过 → VERIFY_PASS → REPORT
  如果仍然失败 → 标记为MANUAL_INTERVENTION → REPORT附手动修复指南
```

## 七、错误处理与日志

### 7.1 四级错误处理

#### 场景A: 权限不足

| 维度 | 内容 |
|------|------|
| 触发条件 | 非管理员运行修复模式 |
| 检测时机 | CHECK_ADMIN |
| 诊断阶段 | 非管理员不影响诊断，仅`netsh wfp show filters`需管理员(无权限则自动跳过) |
| 修复阶段 | 提示用户以管理员重新运行，提供两种选择：①自动提权重启(`Start-Process powershell -Verb RunAs`) ②仅运行诊断并输出修复建议(用户手动执行) |
| 退出码 | 4 |

#### 场景B: 命令超时

| 维度 | 内容 |
|------|------|
| 触发条件 | 模块超过预设时间阈值 |
| 诊断阶段 | 单模块超时→标记TIMEOUT, Stop-Job; 依赖下游标记SKIP/SKIP_BY_DEPENDENCY |
| 修复阶段 | 修复命令超时→重试1次(超时值翻倍); 再次超时→标记TIMEOUT, 跳过、并标记下游SKIP_BY_DEPENDENCY |
| 恢复 | 超时模块在VERIFY阶段作为重点验证项回诊 |

#### 场景C: 部分失败

| 维度 | 内容 |
|------|------|
| 触发条件 | 模块子任务部分成功、部分失败 |
| 诊断阶段 | 模块标记PASS(附warnings数组记录失败子项) |
| 修复阶段 | 独立步骤间不因某步失败而中止; 有依赖关系的失败则跳过下游(SKIP_BY_DEPENDENCY, P2-7) |
| 分析阶段 | ANALYZE对PARTIAL结果生成保守修复方案(仅确认可修复项) |
| 退出码 | 2 |

#### 场景D: 完全失败

| 维度 | 内容 |
|------|------|
| 触发条件 | 全部诊断模块失败或全部修复步骤失败 |
| 检测 | 检查WMI/CIM服务状态、PowerShell执行策略、.NET Framework完整性、PS LanguageMode |
| 恢复 | 输出详细手动修复指南；提示使用Windows自带"网络重置"功能 |
| 退出码 | 3 |
| 报告 | 附带系统环境快照(OS版本、PS版本、.NET版本、WMI状态、PS LanguageMode、磁盘空间) |

#### 7.2.1 日志架构

**修正(审查 P2-2)：** 子 Job 不直接写日志文件。采用"主线程统一写入"模式：

1. 每个模块(含 Job 内执行)将日志事件存入返回对象的 `LogEvents` 数组

```
                       ▼
```

### 8.2 BAT启动器 (NetAid.bat)

**修正(审查 P3-7)：** 使用 `%TEMP%` 临时BAT包装脚本，将原始路径以无转义方式写入，提高含空格/特殊字符路径的可靠性。

```bat
@echo off
setlocal enabledelayedexpansion

:: ============================================================
::  NetAid.bat — 断网急救箱启动器
::  职责: 权限检测 / UAC提权 / PowerShell策略绕过 / 参数透传
::  编码: ASCII (纯英文, 无中文, 无BOM)
:: ============================================================

set "SCRIPT_DIR=%~dp0"
set "PS1_SCRIPT=%SCRIPT_DIR%NetAid.ps1"

:: 0. 检查PS脚本是否存在
if not exist "%PS1_SCRIPT%" (
    echo [ERROR] NetAid.ps1 not found: %PS1_SCRIPT%
    echo Please ensure NetAid.ps1 is in the same directory as NetAid.bat
    pause
    exit /b 1
)

:: 1. 检测管理员权限
net session >nul 2>&1
if %errorlevel% equ 0 (set "IS_ADMIN=1") else (set "IS_ADMIN=0")

:: 2. 收集传入参数
set "ARGS="
:parse_args
if "%~1"=="" goto args_done
set "ARGS=%ARGS% %1"
shift
goto parse_args
:args_done

:: 3. 非管理员 → VBScript ShellExecute UAC提权 (空格安全版)
if "%IS_ADMIN%"=="0" (
    echo [INFO] Requesting administrator privileges...

    :: 创建临时包装BAT (避免含空格路径的多层引号转义问题, P3-7)
    set "WRAPPER=%TEMP%\NetAid_Wrapper.bat"
    (
        echo @echo off
        echo cd /d "%SCRIPT_DIR%\.."
        echo call "%SCRIPT_DIR%NetAid.bat" %ARGS%
    ) > "%WRAPPER%"

    set "VBS_FILE=%TEMP%\NetAid_Elevate.vbs"
    (
        echo Set UAC = CreateObject^("Shell.Application"^)
        echo UAC.ShellExecute "cmd.exe", "/c ""%WRAPPER%""", "", "runas", 1
    ) > "%VBS_FILE%"
    cscript //nologo "%VBS_FILE%" >nul 2>&1
    del "%VBS_FILE%" >nul 2>&1
    del "%WRAPPER%" >nul 2>&1
    exit /b %errorlevel%
)

:: 4. 已是管理员: 设置标题并调用PS
title NetAid - Network First Aid

powershell.exe -NoProfile -ExecutionPolicy Bypass -Command ^
    "[Console]::OutputEncoding = [System.Text.Encoding]::UTF8; ^
     $OutputEncoding = [System.Text.Encoding]::UTF8; ^
     & '%PS1_SCRIPT%' %ARGS%; ^
     exit $LASTEXITCODE"

set "EXIT_CODE=%errorlevel%"
if "%EXIT_CODE%" neq "0" (
    echo.
    echo [NetAid] Exit code: %EXIT_CODE%
)

exit /b %EXIT_CODE%
```

**架构决策说明：**

| 方案 | 方式 | 优缺点 |
|------|------|--------|
| **VBScript ShellExecute("runas")** (采用) | 纯Windows原生，零依赖，UAC对话框最干净 | 不依赖PS已可用 |
| 临时BAT包装(新增P3-7) | 将原始路径写入`%TEMP%`临时包装BAT | 解决含空格/特殊字符路径的ShellExecute可靠性问题 |
| PowerShell Start-Process -Verb RunAs (备选) | PS自举 | 需PS可用，参数含空格转义复杂 |
| runas命令 (不采用) | 需要目标用户明文密码 | 不适合UAC提权场景 |

### 9.3 关键的实现修正备忘

以下为审查中发现的需在编码阶段特别注意的API兼容性/实现陷阱：

| 编号 | 陷阱 | 替代方案/修正 |
|------|------|-------------|
| A1 | `Resolve-DnsName -DnsOnly -QuickTimeout` 不存在于PS5.1 | `Start-Job` + `Wait-Job -Timeout 2` + `[System.Net.Dns]::GetHostEntry()` |
| A2 | `Start-Job` ScriptBlock 无法访问 `$global:DiagCache` | Phase间屏障收集 → `-ArgumentList` 传入数据切片 |
| A3 | `netsh wfp show filters file=-` 部分版本不支持 | 改用 `netsh wfp show filters \| Out-String` |
| A4 | `arp -d *` 不可靠 | 改用 `netsh interface ip delete arpcache` |
| A5 | `Get-WinEvent -FilterXPath` 非英文Windows本地化问题 | 批量替换为 `-FilterHashtable @{LogName=...; Id=...; StartTime=...}` |
| A6 | `netsh int ipv4/ipv6 reset` 与 `netsh int ip reset` 冗余 | 移除 ipv4/ipv6 reset |
| A7 | `ipconfig /registerdns` 用途语义错误(DNS注册并非DNS客户端解析) | 从默认修复链移出，保留为可选高级选项 |
| A8 | BAT路径四次引号转义不可靠 | 使用 `%TEMP%` 临时包装BAT间接调用 |
